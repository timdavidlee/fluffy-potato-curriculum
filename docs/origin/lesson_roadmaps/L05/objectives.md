## Where this lesson sits

L05 is the **third graph lesson**. [L03](../L03/objectives.md) wrapped a *single* LLM call as a typed node; [L04](../L04/objectives.md) wired **several** nodes into a fixed sequence where the developer hard-codes what runs next. L05 adds the one remaining primitive needed before the course can talk about agents: the **conditional edge** — a transition chosen at *runtime* by a routing function that inspects state and returns the name of the next node. With L05, the **L03–L05 graph ramp is complete**: single node → fixed sequence → runtime-chosen branch, all still without tools (L07) and without an agent loop (L10/L11).

The through-line inherited from L04 is the industry **workflow vs. agent** distinction: a workflow runs through code paths the *developer* laid out; an agent lets the *model* direct its own path. L04 opened this contrast with the acyclic/cyclic distinction (a workflow never loops back). L05 **sharpens it** on the axis that actually matters for routing: **who decides which branch runs**. In L05 the decider is always something the developer owns — a computed value, a model's *classification label* written to state, or *direct user input* — and never the model choosing to call a tool. That last case is the agent, and it is not taught until **[L11](../L11/objectives.md)**. L05 is the last stop before the course leaves developer-controlled control flow behind for good (L06 CoT, L07 tools, L10 the hand-rolled loop, L11 the graph-shaped agent) — so this lesson's central job is making the **L05-vs-L11 line** — *you own the branch here; the model owns it there* — completely unambiguous before students meet an agent that blurs it.

Because L05, like L04, sits **before** the tool-calling lessons, its nodes call the model only to *classify* or *extract a label* — never to emit a tool call. A `classify`/`route` node produces a small structured value (a category, a boolean, a choice) that a routing function reads; the model never picks the next node itself.

L05 continues L04's framework choice: nodes call the native LangChain **`ChatAnthropic`** client, not the hand-rolled `potato_llm` seam. No new framework story here — the departure was already made and explained in L04; L05 just keeps using it.

## Prerequisites

Students arriving at L05 should already be able to:

- Build and run a **multi-node sequential graph** (L04) — a typed state schema, several nodes wired with **fixed edges** (`add_edge`), `compile()`, `invoke()`. L05 is "the same graph-building skill, plus one new edge type."
- Distinguish a **node** from an **edge** and explain that a node may call the model while the developer still owns what runs next (L04's headline framing) — L05 depends on this distinction being solid, because it is about to complicate "what runs next."
- Request **structured output** by prompt instruction (L02) — the classifier/router node in L05 needs a small, reliably-parseable label (e.g. a category string) written into state; this is the same structured-output discipline from L02, not tool-use-as-schema (that is L07's territory, deferred further still).
- Reason about **cost and latency** per call (L01) and the **per-node model mixing mechanism** demonstrated in L04 (a light node on Haiku, a heavy node on Sonnet) — L05's classifier is the canonical "cheap model, just needs a label" case and reuses this mechanism rather than re-teaching it.

L05 does **not** prerequisite tracing or the agent loop — both come **later** (tracing at **[L12](../L12/objectives.md)**, the loop at **[L10](../L10/objectives.md)**). Per the same reorder that affected L04, Langfuse stays a one-line **forward reference to L12**, not a dependency. **Decided:** "watch which branch ran" uses **`graph.stream(stream_mode="updates")`** — the built-in run inspection from L03/L04 — which surfaces exactly the node(s) that fired, so only the chosen branch appears. This is the same call threaded L03 → L04 → L05 → L10 (not print statements, not a Mermaid highlight), so students meet one run-inspection tool, not four.

## Learning objectives

By the end of L05, a student should be able to:

1. **Distinguish a fixed edge from a conditional edge and explain who decides.** Concretely:
   - State the difference precisely: a **fixed edge** (`A → B`, always, wired with `add_edge`) versus a **conditional edge** (a **routing function** that reads **state** at runtime and returns the name of the next node, wired with `add_conditional_edges`). L04 used only fixed edges; L05's entire addition is the conditional kind.
   - Name the three things a conditional edge's routing function is allowed to read in this lesson — **derived/computed data** already in state, a **model classification result** (a label a node wrote to state), or **direct user input** (a value supplied as part of the initial state) — and the one thing it is never allowed to read: whether **the model asked for a tool**. Say the L05-vs-L11 line out loud every time: *here you (the developer) own the branch; in an agent, the model owns it.*
   - Trace a conditional edge end to end: a node writes a value into state (a `category` field, a `user_choice` field, a boolean flag) → the routing function reads that field → `add_conditional_edges` maps the routing function's return value to the matching next node. Emphasize that the routing function itself is *plain Python* — an `if`/`match` over state — not a model call.

2. **Build a router/switch: an entry classifier that fans out to specialized branches.** Concretely:
   - Build the classic **routing** workflow shape: a `classify` (or `route`) entry node that calls the model (`ChatAnthropic`) and writes a **label** into state, followed by a conditional edge that sends execution down one of several **specialized branch nodes** (e.g. *billing / technical / general*), which converge to a single exit (`END`).
   - Use **per-node model mixing** deliberately here, reusing the mechanism from L04 rather than re-deriving it: the `classify` node — which only needs to emit a short label — runs on the **cheap, fast model (Haiku 4.5)**; the specialized branch nodes — which do the real reasoning work — run on the **capable model (Sonnet 4.6)**. This is a **mechanism** demo (a graph lets each node bind its own model); the full mixed-model **decision framework** (capability/latency/cost axes, budgets) is **[L14](../L14/objectives.md)**'s job — forward-reference it, do not teach it here.
   - Demonstrate **determinism**: re-run the same input through the router and confirm the same branch executes every time. This is the workhorse proof that a conditional edge, even though it is "runtime-chosen," is not random or model-driven in the agentic sense — it is a deterministic function of state.
   - Handle the **ambiguous/uncertain classification** case explicitly: decide what the router does when the label doesn't cleanly match a branch (a default/`general` fallback branch is the recommended shape) — a router needs a defined behavior for every possible label, not just the expected ones.

3. **Build a conditional edge that branches on direct user input, with no model in the decision.** Concretely:
   - Build a version of the same router shape where the branch is chosen by a value the **user** supplied directly — a menu choice, a form field, a typed command — rather than by a model classification: `if state["user_choice"] == "escalate": return "human_review"`. In L05 this user input arrives **as part of the initial state** (or is collected synchronously before the run starts); the graph then runs straight through. The *interactive* version — a graph that **pauses mid-run to ask** the user and resumes on the answer — needs LangGraph's `interrupt` plus a checkpointer and is out of scope here, forward-referenced to **[L17](../L17/objectives.md)** (human-in-the-loop).
   - Put **model-classification routing** (objective 2) and **user-input routing** (this objective) side by side: *same graph shape — an entry point, a conditional edge, converging branches — different decider.* Land the point that this is the **purest possible contrast with an agent**: with user-input routing, **no model is involved in the routing decision at all** — the user picked the path, the developer wired the options. If a model *is* involved (objective 2's classifier), it produced a label the developer's code then branched on — still not the model choosing the edge.
   - Build one **combined** shape: route on the user's choice first, then run a model-driven node *inside* the chosen branch. Articulate the rule this demonstrates: the **user (or developer logic) owns the edge; the model can still do work inside a node.** This is the general pattern for most real conditional workflows — routing decisions and model reasoning coexist, but they are not the same thing, and only one of them is "the branch."

4. **Contrast developer-controlled branching (L05) with model-driven control (the agent loop, L11) and state the closing argument of the workflow-vs-agent arc.** Concretely:
   - Restate the workflow-vs-agent distinction from L04, now sharpened to the branching case: a **workflow's** conditional edges are decided by developer-owned inputs to state (derived data, a model *label*, direct user input); an **agent's** conditional edges are decided by whether the **model itself** chose to call a tool, typically inside a **cycle** (the model can be routed back to itself). L05 never builds a cycle — every graph in this lesson is still a DAG, still acyclic, still a workflow, no matter how many branches it has.
   - Explain precisely *what changes* between L05 and L11: L05's conditional edge asks "what does **state** say?" (a value the developer put there); L11's conditional edge asks "did the **model's last message** contain a `tool_use`?" (a value the model produced *as its own decision*, not one the developer wrote for it). Same LangGraph mechanism (`add_conditional_edges`, a routing function), completely different **source of authority** over the path.
   - Reason about **when developer-controlled branching is enough** versus when it isn't: a router/switch is the right tool whenever the *set of possible paths is known in advance* and picking among them is a classification, lookup, or user choice — cheap, predictable, testable. Reach for an agent only when the model must decide *which and how many* steps to take, and that set of steps isn't enumerable ahead of time. Name the common failure mode: building an agent to do what a two-branch router would do just as well, at a fraction of the cost and unpredictability.
   - Close the arc explicitly: "L04 showed you a graph with no branches; L05 showed you a graph with developer-owned branches; L11 shows you a graph whose branches the model owns. Same primitives throughout — `StateGraph`, nodes, edges, conditional edges, state, reducers — the only thing that ever changes is *who decides*."

## What a routing graph is (vocabulary the lecture must establish)

Reuses L04's vocabulary verbatim (graph, node, edge, state, reducer, entry point/END, DAG, workflow vs. agent) and adds:

- **Conditional edge** — a runtime-chosen transition: a routing function reads **state** and returns the next node's name, wired with `add_conditional_edges`. **In L05 the routing function reads developer-owned inputs to state — derived data, a model classification label, or direct user input — never whether the model asked for a tool; in L11 it reads exactly that.**
- **Routing function** — the plain Python function (an `if`/`match`, not a model call) that implements a conditional edge's decision. Takes the current state, returns a node name (or a key mapped to a node name).
- **Router / switch** — the recurring shape: one entry node (often a classifier) whose conditional edge fans out to several specialized branch nodes that converge to a single exit. The workhorse pattern of this lesson.
- **Classification label** — a small structured value (a category, a boolean, an enum-like string) a node writes into state for a routing function to read. Produced by a model call using L02's structured-output-by-instruction discipline — not a tool call, not a schema-enforced object.
- **User-input branching** — a conditional edge whose routing function branches on a value the *user* supplied (a choice, a form field, a command) rather than on a model output — arriving in the **initial state** in L05 (no mid-run pause; that is L17's `interrupt`-based variant).
- **Fallback / default branch** — the node a router sends execution to when the classification label doesn't match any known branch. A router must define behavior for the unmatched case, not just the expected ones.
- **Decider** — informal shorthand this lesson uses repeatedly: *what* or *who* supplies the value a conditional edge branches on (derived data / a model label / user input, in L05; the model's own tool-call decision, in L11). "Same mechanism, different decider" is the lesson's refrain.

## Main points the lecture should land

- **The conditional edge is the one new primitive this lesson adds — everything else is L04, reused.** State schema, nodes, fixed edges, `compile`/`invoke`, reducers: all unchanged. L05 is deliberately narrow so the one new idea is unmissable.
- **"Runtime-chosen" does not mean "unpredictable."** A conditional edge is still a deterministic function of state: same state in, same branch out, every time. Determinism is what separates a workflow's routing from an agent's — re-running is a proof, not a triviality.
- **The decider can be derived data, a model label, or a human choice — and those are three different things worth telling apart.** All three are still "the developer owns the branch," because in every case the *routing function*, written by the developer, is what reads the value and picks the edge. The model (if involved) only ever produces a label; it never picks the node.
- **User-input branching is the purest possible contrast with an agent.** No model is anywhere near the routing decision. Keep coming back to this case whenever the model-classifier case risks blurring the line — it's the cleanest anchor for "the model didn't choose this, I did."
- **A router needs a defined behavior for the unexpected label, not just the expected ones.** Treat the fallback/default branch as a first-class part of the design, not an afterthought — this is a habit L05 should install early, before students meet messier real-world classifiers.
- **The workflow-vs-agent arc closes here, not by building an agent, but by naming precisely what an agent would change.** L05 does not build a cycle and does not let the model choose an edge — it *defines*, crisply, the exact thing L11 is going to do differently, so that when students meet L11 the new idea is a single, well-anticipated change rather than a surprise.
- **Per-node model mixing is a natural fit for routing, and this is the second time students see it (reinforcement, not new material).** A cheap model is enough to produce a label; the branches that actually need judgment get the capable model. The lesson reuses L04's mechanism rather than re-explaining it — one line of recall, then move on.

## Common student confusions to watch for

- *"The router is deciding, so the model must be deciding."* No — the **routing function** decides, by reading a value in state. If that value came from a model call, the model produced a *label*; the developer's plain-Python routing function is what turned the label into a branch choice. Reuse L04's framing: same mechanism, different decider, name which one every time.
- *"A conditional edge in L05 and a conditional edge in L11 are basically the same thing, so what's the big deal?"* Mechanically, yes — same LangGraph API (`add_conditional_edges`), same shape. The entire lesson is about the one thing that's different: *what the routing function is allowed to look at.* State-owned-by-developer versus the-model's-own-tool-call-decision is the whole distinction, and it is easy to let it collapse if the mechanical similarity is emphasized without the decider distinction.
- *"User-input branching means the graph is asking the user something."* Not in L05 — the user's value is already sitting in the initial state before the graph runs; nothing is asked *during* the run. The mid-run ask-and-resume version needs `interrupt` + a checkpointer and belongs to L17.
- *"If there's no obvious matching branch, just let it error."* An unhandled label is a design gap, not an edge case to skip — the lecture should insist on a named fallback/default branch as part of "the router is done," not an optional extra.
- *"More branches = closer to being an agent."* No — branch *count* is irrelevant to the workflow/agent distinction. A ten-way router is still 100% a workflow if the developer wired every branch; a one-decision agent loop (call one tool or stop) is still an agent because the *model* made that one choice. Depth/width of branching is not the axis; who decides is.
- *"This is basically a big if/else, why use a graph?"* Fair, and worth saying honestly: for a single router, a graph is close to break-even over a plain `if`/`elif` chain in Python. The payoff (inspectable/traceable control flow, mixing this router into a larger pipeline, converging branches cleanly, later composing with agent primitives at L11) shows up as the system grows — name the honest trade rather than oversell it, matching L04's stance on the same question.

## Bridge to L11

**[L11](../L11/objectives.md)** (Shallow agents in LangGraph) is where the workflow-vs-agent arc — opened at L04, sharpened here — finally resolves. L11 reuses every primitive from L03–L05 (`StateGraph`, nodes, fixed edges, conditional edges, typed state, reducers, compile/invoke) and changes exactly one thing: the conditional edge's routing function stops reading developer-owned state and starts reading **whether the model's last message contains a `tool_use`** — and that conditional edge is wired into a **cycle** (routing back to the model node), not just a one-shot branch. L05 does not need to preview the cycle mechanically; its job is to have made the "who decides" question so automatic that L11's change reads as *exactly one new thing*, not a new mental model. Per L04's convention, **L11 keeps its own self-contained primitives intro** rather than depending on L05 having taught them, so a student who starts at L11 still gets nodes/edges/state from scratch — the two should agree on vocabulary verbatim (the shared list above) so the overlap reads as reinforcement.
