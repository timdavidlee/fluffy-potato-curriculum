# L11: Explicit graphs & workflows in LangGraph (deterministic DAGs)

> Parent design doc: [CURRICULUM_PRD.md](../../CURRICULUM_PRD.md) (lesson-plan row L11).
> Folder conventions: [docs/origin/CLAUDE.md](../../CLAUDE.md).
> Preceding lesson: L10 Choosing model power for the task (roadmap not yet written; in the mini cut the previous taught lesson is [L09 Evaluation — first pass](../L09/objectives.md)). Following lesson: [L12 Shallow agents in LangGraph](../L12/objectives.md).

## Where this lesson sits

This is the **first LangGraph lesson**, and it is deliberately placed *before* the agent. Up to here students have written control flow as plain Python: a single call (L01–L05), a single tool round-trip (L04), and a hand-rolled **model → tool → model loop** (L07) whose path the *Python `while`/`if`* decided. L11 introduces LangGraph not by building an agent, but by building a **workflow** — a fixed, **directed acyclic graph (DAG)** of explicit nodes where the **developer wires the path** and the model never decides what runs next. L12 then takes the *same* primitives and adds the one thing that turns a workflow into an agent: a cycle.

The framing the lesson exists to install is the industry's **workflow vs. agent** distinction (as in Anthropic's *Building Effective Agents*): a **workflow** runs through predefined code paths the developer laid out; an **agent** lets the *model* direct its own process — choosing tools and looping until it decides it's done. LangGraph expresses **both**. Teaching the deterministic workflow first means that when L12 hands control of the path to the model, students see exactly what changed (an acyclic graph gained a cycle; a developer-owned edge became a model-owned one) instead of meeting graphs and agency fused into one intimidating idea.

Concretely, L11 builds two canonical workflow shapes — **prompt chaining** (decompose a task into a fixed sequence of steps: e.g. *extract → transform → summarize*) and **routing** (classify the input, then branch to one of several specialized paths) — on LangGraph's `StateGraph`. The model is used *inside* nodes (a node may call Claude to classify or summarize), but the **edges are fixed or chosen by developer logic over the state**, never by the model emitting a tool call. That is the whole point: in a workflow you own the flow; in an agent (L12) the model owns it.

This is also the lesson where the course **first reaches for a framework** and, with it, the framework's own model client. Per the project's provider decision, from L11 onward the LangGraph lessons call the **native LangChain `ChatAnthropic`** model directly inside graph nodes rather than the hand-rolled `potato_llm` seam used in L01–L09 — and the lecture names that departure out loud ("frameworks bring their own client abstraction"). **Decided:** the "native client, not the seam" departure starts at L11 (the new first framework lesson); the provider-abstraction memory has been updated to say the framework arc begins at L11.

## Prerequisites

Students arriving at L11 should already be able to:

- Send a chat completion and request **structured output** by prompt instruction (L02) — a routing workflow's classifier node returns a structured label the graph branches on.
- Read a **trace** of a run and reason about it (L08). LangGraph emits its own execution trace, and L11 routes that trace to the **same self-hosted Langfuse** instance students met in L08 — so "watch the workflow run" reuses a skill they already have.
- Reason about **cost and latency** per call (L01) — a prompt-chaining workflow makes *multiple* model calls in sequence, and the per-step cost is exactly what L11's "why decompose?" discussion weighs.
- Read and write basic Python control flow (`if`/`for`/functions) and typed data structures — L11 turns that control flow into *graph data* (nodes and edges), so students must recognize the `if`/`then` they're replacing.

L11 does **not** require the L07 agent loop as a hard prerequisite — a deterministic workflow has no agent loop — but it sits after L07 in the sequence, and the contrast ("the L07 loop let the *model* keep going; this workflow does not") is a useful reference. **Decided:** L11 references L07 for contrast but does not depend on it — a workflow is conceptually upstream of an agent and stands on its own.

## Learning objectives

By the end of L11, a student should be able to:

1. **Model a multi-step task as a directed acyclic graph of explicit nodes.** Concretely:
   - Use LangGraph's `StateGraph` builder to define a graph: declare a typed **state** schema, add **nodes** (each a function that reads state and returns a state update), set the **entry point**, wire **edges**, and **compile** the graph into a runnable, then **invoke** it on an input.
   - Implement a node as a plain typed function — e.g. an `extract` node that calls Claude (`ChatAnthropic`) and writes its result into state; the node does one unit of work and hands back an update, nothing more.
   - **Bind a different model per node.** Because each node is its *own* independent model call, each can construct its own `ChatAnthropic(model=...)` — a **cheap, fast model** (Claude **Haiku 4.5**) for the light steps (classify, route, extract a field) and a **capable model** (Claude **Sonnet 4.6**) for the heavy reasoning steps (draft, analyze). Show a workflow whose `classify` node runs Haiku and whose `draft` node runs Sonnet, and have students read the trace to see each span's model. This is the **mechanism** of mixed-model design — *which* model each step deserves (the capability / latency / cost trade-off axes, and budgets) is **L10's** topic (Choosing model power); L11 just makes concrete that a graph lets you mix models per node. **Decided:** L11 demonstrates the *mechanism* (per-node `ChatAnthropic(model=...)`) with a light cost/latency aside read off the trace; the full mixed-model *decision framework* ("small model for routing/classification, capable model for reasoning"; capability/latency/cost axes; budgets) is L10's — the two reinforce, not re-teach. In the mini cut (L10 dropped) this is students' first mixed-model exposure, kept light but present.
   - Articulate *why a graph instead of a function that just calls things in order*: the graph makes control flow **inspectable data** (nodes and edges you can list, visualize, and trace) rather than imperative statements buried in Python. Name the honest trade: for a strictly linear three-step task the graph is near break-even; its value shows up with branching, visualization, shared state, and built-in tracing.
   - Keep the graph **acyclic** — every edge moves forward; there is no edge that returns to an earlier node. State that the absence of a back-edge is exactly what makes this a *workflow* and not yet an *agent*.

2. **Wire fixed edges and shared state through the graph.** Concretely:
   - Distinguish a **fixed edge** (`A → B`, always) from a **conditional edge** (a routing function inspects the **state** and returns the name of the next node). Emphasize the critical L11-vs-L12 point: in L11 the routing function branches on **developer-owned inputs to the state** — computed/derived data, a model *classification result*, or **direct user input** (e.g. `if state["category"] == "refund": return "refund_node"`) — **never** on the model emitting a tool call. That last case is L12.
   - **Branch on user input.** The purest form of "you wire the flow" is a conditional edge whose routing function reads a value the **user** supplied — a menu choice, a form field, a yes/no, a typed command — and routes deterministically: `if state["user_choice"] == "escalate": return "human_review"`. Here **no model is involved in the routing decision at all**: the *user* picked the path, the *developer* wired the options. It's the sharpest possible contrast with an agent (where the model picks the path), and a natural mid-flow decision point in a workflow. In L11 the user input arrives **as part of the initial state** (or a synchronous prompt answered before the run) so the graph runs straight through; the *interactive* version — a graph that **pauses to ask** the user and resumes on their answer — needs LangGraph's `interrupt` + a checkpointer and is a forward pointer to L15 (human-in-the-loop). **Decided:** L11 covers user-input branching only in the "input is in the initial state" form (checkpointer-free); the interactive pause/resume `interrupt` version is L15's territory.
   - Define the **state** as a typed object (a `TypedDict` / annotated state, matching the repo's typed style) threaded through every node, and explain how a node **returns an update** that LangGraph merges into state via per-field **reducers** (e.g. an `add`/append reducer for an accumulating list). This is the same state/reducer machinery L12 reuses for an agent's message history — introduced here on a simpler, acyclic graph.
   - Distinguish **what belongs in state** (data that flows between nodes — the extracted fields, the classification, intermediate results) from **what doesn't** (the model client, configuration — dependencies wired in at build time, not state).

3. **Build a deterministic prompt-chaining / routing workflow where the developer controls the path.** Concretely:
   - Build a **prompt-chaining** workflow: a fixed sequence of nodes where each step's output feeds the next (e.g. *parse a support ticket → draft a reply → check the reply against a policy*), each node a separate model call with a focused prompt. Explain *why decompose* — smaller focused prompts are more reliable and individually testable than one mega-prompt, at the cost of more calls (the L01 cost trade).
   - Build a **routing** workflow: an entry classifier node labels the input, and a conditional edge sends it down one of several specialized branches (e.g. *billing / technical / general* each with its own prompt), which then converge to an exit. The path is chosen by **the classification result in state**, deterministically — re-running the same input takes the same path. This is also the natural place to show **per-node model selection**: run the cheap classifier node on **Haiku 4.5** and the specialized branch nodes on **Sonnet 4.6** — a small model where you just need a label, a capable model where you need quality. Read the Langfuse trace and point out that each span shows its own model (and its own cost).
   - Build a **user-input-branching** workflow: the path is chosen by **user input** rather than a model classifier — e.g. a triage graph where the user selects a request type (or supplies a structured choice) and a conditional edge routes to the matching branch. Put it head-to-head with the model-classifier routing above: *same graph shape, different decider* — the **user** chooses the path, not the model. Both are still deterministic workflows because the developer wired every branch. The takeaway to land: "routing" in a workflow can be driven by **derived data, a model label, or a human choice** — none of which is the model driving a *loop* (that's the agent, L12). A natural combined example: route on the user's choice first, then run a model-driven *node* inside the chosen branch — user owns the edge, model works inside the node.
   - Run both workflows and **trace** them: route the execution to the shared **Langfuse** instance (L08) and read the spans — a linear chain of GENERATION spans for prompt-chaining; the chosen branch for routing — confirming the path was developer-determined.
   - Optionally **carry the L09 eval discipline forward**: a deterministic workflow is the *easiest* thing to evaluate (same input → same path), so a tiny eval set over the routing classifier is a natural, cheap reinforcement of "evaluate every thing you build." **Decided:** include one short optional eval-the-workflow beat reusing `common/evals.py` — it reinforces L09 and shows workflows are trivially testable (same input → same path), which is half the point of preferring them.

4. **Contrast a workflow (you wire the flow) with an agent (the model drives the flow).** Concretely:
   - State the distinction crisply and reuse it verbatim into L12: a **workflow** is a graph whose path is fixed or chosen by *developer logic* (acyclic, predictable, the model lives inside nodes); an **agent** is a graph whose path is chosen by the *model*, typically via a **cycle** that loops model → tools → model until the model stops (L12's shallow agent, L07's hand-rolled loop).
   - Reason about *when to use which*: prefer a **workflow** when the task has a known shape — it is more predictable, cheaper, lower-latency, and far easier to test and trace; reach for an **agent** when the steps can't be known in advance and the model genuinely needs to decide its own path. Name the common failure mode: reaching for an agent when a workflow would do (more cost, less predictability, harder to debug).
   - Point forward precisely: L12 keeps everything from this lesson — nodes, edges, typed state, reducers — and changes exactly one thing: it adds a **conditional edge that loops back to the model**, handing the model control of the path. That single back-edge is the line between this lesson and the next.

## What a workflow graph is (vocabulary the lecture must establish)

Define these explicitly and reuse them verbatim into L12:

- **Graph** — a task expressed as nodes connected by edges, compiled into a runnable. The framework's structured replacement for imperative control flow.
- **Node** — a unit of work: a typed function that reads state and returns a state update. May call the model (`ChatAnthropic`) internally — and **each node may bind its own model** (a cheap model for light steps, a capable model for heavy ones), since each node is an independent call.
- **Edge** — a fixed transition between nodes (`A → B`, always).
- **Conditional edge** — a runtime-chosen transition: a routing function reads **state** and returns the next node's name. **In L11 the choice is made by developer-owned inputs to the state — derived data, a model *classification result*, or *direct user input* — never by the model deciding to call a tool; in L12 the model makes the choice.**
- **User-input branching** — a conditional edge whose routing function branches on a value the *user* supplied (a choice, form field, command) rather than on a model output. The purest "you wire the flow": no model in the routing decision. (The *interactive* pause-and-ask-the-user variant is L15.)
- **State** — the shared, typed object passed through every node; carries the data that flows between steps.
- **Reducer** — the rule that merges a node's returned update into state per field (overwrite, or append for accumulating fields). Same mechanism L12 reuses for message history.
- **Entry point / END** — where execution starts and the sentinel where it stops.
- **DAG (directed acyclic graph)** — a graph with no back-edges: every path moves forward to `END`. The defining shape of a *workflow*.
- **Workflow vs. agent** — workflow = developer wires the (acyclic) path; agent = the model drives the (cyclic) path. The lesson's headline contrast.

## Main points the lecture should land

- **LangGraph is not just for agents.** A graph can be a plain deterministic **workflow** — no agency required. Introducing graphs this way separates "learn the framework's primitives" from "let the model drive," which is two hard ideas, not one.
- **In a workflow, the model lives *inside* the nodes; the developer owns the edges.** The model still does the smart per-step work (classify, summarize, draft), but *what runs next* is decided by code, not by the model. In an agent (L12) the model owns the edges too. Say this contrast early and often.
- **A graph turns control flow into inspectable data.** Nodes and edges can be listed, drawn, traced, and reasoned about — unlike `if`/`while` buried in Python. For a single linear chain that's near break-even; it pays off with branching, visualization, shared state, and tracing.
- **Determinism is a feature, not a limitation.** A workflow takes the same path on the same input: it's predictable, cheaper, lower-latency, and trivially testable (which is why the L09 eval discipline fits it so neatly). Most production "AI features" are workflows, not agents.
- **Prompt chaining and routing are the two workhorse patterns.** Decompose a task into fixed focused steps (chaining), or classify-then-branch (routing). Both keep the path in the developer's hands and each prompt small and reliable.
- **A branch can be decided by data, a model label, or the user — and user input is the purest "you wire the flow."** A conditional edge can route on derived data, on a model *classification*, or on **direct user input** (a choice, a form field) with no model in the routing decision at all. Most real "AI workflows" mix model-classified routing with plain user-input routing. It's the cleanest contrast with an agent: here the *user/developer* decides the path; in L12 the *model* does. (The interactive pause-and-ask-the-user variant needs a checkpointer and is L15.)
- **Each node can use its own model.** A node is an independent call, so a graph can mix models — a cheap, fast model (Haiku 4.5) for routing/classification/extraction, a capable model (Sonnet 4.6) for the reasoning steps. This is the *mechanism* of mixed-model design; the *when/why* (capability vs. latency vs. cost, budgets) is L10's job. The trace makes it tangible: each span shows its own model and cost.
- **State and reducers are introduced here, reused for agents.** The typed state threaded through nodes, and the reducer that merges updates, are the *same* machinery L12 uses for an agent's message history — learned first on a simpler acyclic graph.
- **The workflow→agent line is a single back-edge.** Everything students build here carries into L12 unchanged; the only addition is a conditional edge that loops to the model. Framing it this way makes L12 feel like one small step, not a new world.

## Common student confusions to watch for

- *"A LangGraph graph is an agent."* No — a graph is just a wired set of nodes. It's an **agent** only when the **model** drives the path (usually via a loop). L11's graphs are workflows: the developer drives the path.
- *"A conditional edge means the model is deciding."* Not in L11. Here a routing function branches on **state** the developer set — derived data, a model classification *label*, or **direct user input**. In L12, the routing function branches on whether the **model** asked for a tool. Same mechanism, different decider — call out which it is every time.
- *"Branching always means the model is choosing."* No. In a workflow the branch can be decided by developer logic, a model *classification* (a label in state), or **direct user input** (a menu choice). Only in an agent (L12) does the model choose the path by deciding to call a tool and loop. A user-input branch has *no model* in the routing decision at all.
- *"Workflows are the weak/beginner version of agents."* No — they're a different tool with real advantages (predictability, cost, latency, testability). The engineering skill is choosing the simplest shape that solves the task, and that's usually a workflow.
- *"If the model is involved, it must be an agent."* The model is involved in a workflow too — inside the nodes. Agency is about who controls the *path*, not whether a model is called.
- *"Decomposing into multiple prompts is wasteful — one big prompt is cheaper."* Sometimes cheaper in tokens, usually worse in reliability and much harder to test or fix. Chaining trades some cost for focused, individually-verifiable steps.
- *"My workflow needs a loop."* If it genuinely needs the model to keep going until *it* decides to stop, that's an agent (L12) — don't force a cycle into a workflow. If it just needs to repeat a fixed number of times over known data, that's still a (possibly unrolled) workflow.

## Bridge to L12

L12 (Shallow agents in LangGraph) is the direct sequel and the payoff of teaching workflows first. It reuses **every** primitive from this lesson — `StateGraph`, nodes, edges, typed state, reducers, `compile`/`invoke`, and the Langfuse tracing hookup — and makes exactly one change: it adds a **conditional edge that loops back to the model**, so the *model* decides whether to call a tool or finish. That single back-edge converts the acyclic **workflow** into the cyclic, model-driven **agent** (the same model → tool → model loop students hand-rolled in L07, now as a graph).

Per the project decision, **L12 keeps its own self-contained primitives intro** rather than depending on L11 to have taught them — so a student who only takes the agent lesson still gets nodes/edges/state from scratch. The cost is some intentional overlap between L11 and L12 on the primitives; the benefit is each lesson stands alone. The two should agree on vocabulary verbatim (the list above) so the overlap reads as reinforcement, not contradiction. **Decided:** L11 and L12 both teach nodes/edges/state (intentional overlap, "keep both self-contained"); L12's intro explicitly calls back to L11's workflow-vs-agent framing ("you saw the acyclic version in L11; here's the cycle") so the overlap reads as deliberate reinforcement.
