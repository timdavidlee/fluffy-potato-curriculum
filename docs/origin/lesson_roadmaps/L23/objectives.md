## Where this lesson sits

In [L22](../L22/objectives.md) students met **the skill**: a named, described, markdown-based bundle of instructions (and optional scripts) an agent loads **just in time**, when the model judges its **description** relevant, instead of carrying it always-on. They hand-built a minimal JIT loader (name+description always visible → body on selection → scripts on execution — **progressive disclosure**), authored one real `SKILL.md`, and learned the core taxonomy: **a tool is *called*, a skill is *read* on demand, the system prompt is *always seen*.** L22 answered *"where should a single capability live?"*

L23 is the sequel to that, on two axes L22 deliberately left open:

1. **Skills are not all the same shape.** L22 authored *one* skill and treated "skill" as a single thing. In practice skills fall into a few recurring **archetypes**, each with a different center of gravity: a recipe that mostly runs a **bundled script** (an API/integration wrapper), a **rubric** of rules or principles the agent applies as judgment (a review/checklist skill), and an **operating-model runbook** that drives a multi-step procedure end to end. Recognizing the archetype tells you how to write the `SKILL.md` — where the real content lives (in a script, in a rule list, in an ordered procedure) and how imperative to be.

2. **Skills compose into systems.** One skill rarely stands alone. L23 teaches the two composition moves that turn a pile of skills into a system: **sequential handoff** (one skill's output is the next skill's input — like this very curriculum's `author-lesson-roadmap` → `generate-materials-from-roadmap` two-stage flow) and a **shared lower-level skill** that several higher-level "operating" skills all invoke (write-once, reuse-many). Once skills call skills, there's a **dependency graph**, and JIT loading now happens *across* that graph — which is where the interesting cost reasoning and the anti-patterns live.

So L23's arc is: **classify** a skill by archetype → **author** each archetype well → **compose** skills two ways → **reason about the resulting dependency graph and its JIT-loading behavior** → **spot the anti-patterns** that make a skill system worse than the pile of prompts it replaced.

Throughout, the runtime is the **[L04](../L04/objectives.md)/[L11](../L11/objectives.md) LangGraph agent** given two skill-loading **tools** (real LangChain tools, in the L07/L08 sense): **`list_skills()`** returns every registered skill's `{name, description}` (discovery), and **`load_skill(name)`** reads that skill's full `SKILL.md` body into context (load). The model *calls `list_skills` to see what's available, then `load_skill` to pull the one it needs.* This is L22's JIT mechanism expressed as agent tools rather than a hand-rolled loader — and it makes composition concrete: "skill A invokes skill B" is literally *A's body instructing the agent to `load_skill("B")`.* <!-- DECIDED (2026-07-01): runtime substrate = the L04/L11 LangGraph agent extended with `list_skills` + `load_skill` LangChain tools (Sonnet 4.6). No new runtime dependency beyond LangGraph, in use since L04. -->

## Prerequisites

Students arriving at L23 should already be able to (all from [L22](../L22/objectives.md) unless noted):

- **Define a skill and place it in the taxonomy** — *read on demand* vs. a tool *called with a schema* vs. the system prompt *always seen* (L22). L23 assumes this is fluent; it is not re-taught, only built on.
- **Author one `SKILL.md`** — YAML frontmatter (`name`, `description`), a markdown instruction body, optional bundled scripts — and understand that **the description is the trigger** the model matches on (L22's load-bearing craft point). L23 extends single-skill authoring to *archetype-appropriate* authoring and to descriptions that must stay distinct *across a set* of skills.
- **Explain just-in-time loading and progressive disclosure** — name+description always in context, body on selection, scripts on execution (L22). L23 scales this from one skill to a graph of them.
- **Decide skill vs. tool vs. system-prompt content** for a single capability (L22). L23 reuses this exact heuristic to catch the "shared skill that's really a tool" anti-pattern.
- **Wire and reason about a tool** — a function with a structured schema, always in context, called mid-reasoning (L07, L08). Needed to draw the skill/tool line precisely when composing.
- **Reason about token cost and the context window** (L01, L07). Composition's payoff and its failure modes are both measured in always-on tokens across the whole skill set.

L23 does **not** require L09 (MCP) or L24 (multi-agent) — both are outside the mini cut. MCP is named only as an aside ("another way to package capability"); multi-agent is named only as the forward bridge. <!-- DECIDED (2026-07-04): L23 stands alone in the mini cut (no L09, no L24) — MCP and multi-agent appear only as named asides / forward-pointers, never as load-bearing prerequisites. -->

## Learning objectives

By the end of L23, a student should be able to:

1. **Classify a skill by archetype, and say why the archetype matters.** Concretely:
   - Name and recognize the archetypes (reuse these labels verbatim through the labs). The organizing idea: **every skill bundles some mix of *instructions*, *scripts*, and *resources*, and the archetype is just which one is the center of gravity** — with instructions splitting by whether they read as *ordered steps* or *judgment criteria*. That gives three **core** archetypes (plus one honorable-mention, below):
     - **API/integration recipe (script-centric).** The skill's real work is done by a **bundled script**; the markdown is a thin wrapper that tells the agent *when* to run it, *how* to call it, and how to read its output. Center of gravity: the script. Example shape: "to fetch an order, run `get_order.py <order_id>` and parse the JSON it prints." Good when the operation is deterministic and better executed than reasoned through.
     - **Review / rubric (principle-centric).** The skill is a **set of rules, principles, or a checklist** the agent applies as *judgment* — no single deterministic answer, no script that could replace it. Center of gravity: the rule list. Example shape: the repo's own [python-style](../../../.claude/rules/python-style.md) / [pytest](../../../.claude/rules/pytest.md) rules, or a `/code-review`-style "check the diff against these principles" skill. Good when the value is consistent application of taste/standards.
     - **Operating-model runbook (procedure-centric).** The skill drives a **multi-step procedure end to end** — an ordered sequence with steps, decisions, and handoffs. Center of gravity: the ordered procedure. Example shape: this repo's `author-lesson-roadmap` skill (read these docs in order → draft → cross-check → report). Good when the value is *doing the steps in the right order every time*.
     - ***(Honorable mention)* Reference / knowledge skill (resource-centric).** The body is **facts to read** — not steps to follow, criteria to judge, or a script to run: a short data-schema glossary, an API field list, a handful of brand or product facts. Center of gravity: the resource. It's the right, cheap tool **when there are only a handful of terms to know** — the facts just sit in-context when the skill loads; *you don't need a whole retrieval subsystem for a dozen definitions*. Once the knowledge outgrows what fits comfortably in-context, that's **RAG's** job ([L21](../L21/objectives.md), full course), not a skill. Taught as an aside — a real, common shape, but not one of the three core authoring exercises.
   - Explain the payoff of classifying first: **the archetype tells you where the content belongs** — push determinism into a *script* (archetype 1), express standards as a *rule list* the agent judges against (archetype 2), write an *imperative ordered procedure* (archetype 3), or just *state the facts* when it's a small reference (the aside). Mis-classifying produces a bad skill (e.g. prose-describing a deterministic step instead of scripting it, or standing up RAG for six definitions).
   - Recognize that archetypes are a **spotting heuristic, not rigid boxes** — a real skill can lean primarily on one while borrowing from another (a runbook that *calls* a script for one step; a review skill that ends by running a formatter; a runbook that opens with a short reference glossary). The point is knowing the *primary* shape so the authoring choices follow.

2. **Author each archetype in the `SKILL.md` format.** Concretely:
   - Produce a well-formed `SKILL.md` for each of the three **core** archetypes — same frontmatter+body format from L22 — with the body *shaped to the archetype*:
     - Script-centric: minimal prose, a clear "when to run it," the exact invocation, and how to interpret output; the deterministic logic lives in the script, not the markdown.
     - Rubric: an explicit, scannable list of rules/principles with enough example to apply each; written as *judgment criteria*, not a linear procedure.
     - Runbook: numbered, imperative, self-contained steps with explicit decision points and a clear done-condition; written *for an agent to follow*, deterministic steps delegated to scripts.
   - Carry L22's craft forward: every archetype still needs a **discriminating description** (the trigger) and instructions written *for an agent, not a human reader*. L23 adds the constraint that descriptions must stay distinct **across the whole set** of skills (see objective 5's *description collisions*).
   - Point at the repo's real skills as worked examples of each archetype rather than inventing formats. <!-- DECIDED (2026-07-01): three CORE archetypes taught and authored — script-centric (API/integration recipe), principle-centric (review/rubric), procedure-centric (operating-model runbook), matching the PRD subgoal. A fourth, **reference/knowledge (resource-centric)**, is taught as a one-line aside (not a core authoring exercise): the right cheap tool for a handful of facts; RAG (L21) only once the knowledge outgrows in-context. Framed via the {instructions | scripts | resources} center-of-gravity model. Vocabulary section carries the industry-term mapping. -->

3. **Compose skills two ways: sequential handoff and a shared lower-level skill.** Concretely:
   - **Sequential handoff** — one skill produces an artifact that a second skill consumes; the pipeline is the capability. Teach it on the repo's real two-stage flow: `author-lesson-roadmap` (stage 1: write the roadmap docs) hands off to `generate-materials-from-roadmap` (stage 2: generate student materials from those docs). Students should be able to identify the **contract at the seam** (what artifact stage 1 must produce for stage 2 to work) and why splitting into two skills beats one mega-skill (each is separately loadable, testable, and re-runnable; JIT-loaded only for the stage you're in).
   - **Shared lower-level skill** — a single small skill that several higher-level "operating" skills all invoke, so a common capability is written once and reused. Example shape: a `render-lesson-links` or `check-lesson-numbering` sub-skill that both an authoring skill and a materials-generation skill call, rather than each re-implementing it. Students should see this as the DRY move for skills and be able to point to the shared node in the graph.
   - Distinguish the two shapes: sequential handoff is **A → B** (output-to-input, a *pipeline*); a shared sub-skill is **A → C ← B** (two callers, one *dependency*). Both create a graph; they create different *shapes* of graph.
   - *One-line aside only (do not teach as a third pattern):* a **conditional/dispatch** shape — one skill routing to *one of several* downstream skills (a branch) — also exists. It is deliberately deferred: routing-to-one-of-N is the **supervisor / orchestration** pattern, taught at the *agent* level in [L24](../L24/objectives.md). Name it and point forward; don't build it here. <!-- DECIDED (2026-07-01): teach exactly TWO composition patterns — (a) sequential handoff, (b) shared lower-level skill — per the PRD subgoal. Conditional/dispatch is PUNTED to L24 as supervisory orchestration (agent-level routing); L23 mentions it in one line as a forward-pointer only. -->

4. **Reason about the skill dependency graph and just-in-time loading across it.** Concretely:
   - Draw the **dependency graph** of a small skill system (nodes = skills, edges = "invokes / hands off to"), using the repo's own skills as the worked example. Identify pipelines (chains), shared nodes (fan-in), and leaves (scripts). Make the edges concrete on the `list_skills`/`load_skill` runtime: an edge *A → B* is literally *skill A's body instructing the agent to `load_skill("B")`* — the graph is the map of possible `load_skill` calls.
   - Extend L22's JIT-loading story from one skill to the graph, realized through the two tools: what's **always-on is just the `list_skills` + `load_skill` tool schemas** — *not* the skills themselves. The agent calls `list_skills` to surface the registry of `{name, description}` only when it's choosing, and each `load_skill` pulls exactly one body along the path taken; a shared sub-skill's body loads *once per use*, not once per caller. Quantify with the L01/L07 cost lens: the skill set's always-on footprint is two tool schemas *regardless of how many skills exist or the graph's shape* — body cost is paid only on the actual path. (This *refines* L22, where the name+description layer sat always-on; routing discovery through `list_skills` pushes even that to just-in-time.) <!-- DECIDED (2026-07-01): teach this refinement of L22's cost model — with `list_skills`, always-on cost is "two tool schemas" (not "N descriptions"), and the N descriptions become a per-`list_skills`-call cost. Frame it explicitly as *refining* L22, not contradicting it — it's the payoff of the tool-based runtime. -->
   - Reason about **depth**: each hop in a handoff or each level of sub-skill nesting is an extra load (and often an extra model turn). Shallow, wide graphs (many independent skills, few hops) load cheaply and fail locally; deep chains concentrate risk — a miss anywhere breaks the pipeline. This is the setup for the *over-chaining* anti-pattern.
   - Connect to the broader picture without re-teaching it: managing what's in a long-running agent's context across many loads is L19's topic (context management, full course); orchestrating whole agents instead of skills is [L24](../L24/objectives.md). Name them; don't teach them here.

5. **Recognize composition anti-patterns.** Concretely, name and diagnose each (reuse these labels):
   - **Over-chaining** — decomposing into so many tiny handoffs that the pipeline is mostly load-overhead and every hop is a place selection can fail. Symptom: a five-skill chain where a two-step skill would do; latency and miss-risk dominate the actual work. Cure: collapse hops; a handoff should exist because the seam is a real reusable/testable boundary, not for tidiness.
   - **A shared "skill" that's really a tool** — the "shared lower-level skill" turns out to be a deterministic operation with structured I/O that every caller invokes the same way. That's L08/L22's definition of a **tool**, not a skill — it should be *called with a schema*, not *read as instructions*. Symptom: the "skill" is a script with a one-line "run this" wrapper and no judgment. Cure: make it a tool (or let the script be the interface) and drop the skill wrapper.
   - **Description collisions** — two or more skills whose descriptions overlap, so the model can't reliably pick the right one; selection degrades across the *whole set*, not just the colliding pair. This is L22's "the description is the trigger" failure, scaled to a system. Symptom: the wrong skill loads, or the model dithers. Cure: make descriptions *mutually discriminating* — each says what it's for *and implicitly what it's not*.
   - Tie all three back to the payoff: composition is only a win if the system is *cheaper to run and easier to reason about* than the monolith it replaced. These anti-patterns are the ways it silently becomes worse.

## Vocabulary the lecture must establish

Define these explicitly and reuse them verbatim through the labs (they build on L22's vocabulary — skill, JIT loading, progressive disclosure, description-as-trigger, always-on context — which are assumed, not redefined):

- **Skill archetype** — the recurring *shape* of a skill, named by its center of gravity (which of *instructions / scripts / resources* dominates). The three core shapes, with the industry terms students will meet elsewhere:
  - **script-centric (API/integration recipe)** — *a.k.a.* tool wrapper, executable skill, action/plugin.
  - **principle-centric (review/rubric)** — *a.k.a.* checklist, style guide, guidelines, policy, *evaluative* playbook.
  - **procedure-centric (operating-model runbook)** — *a.k.a.* runbook, SOP, recipe, *procedural* playbook.
  - (Caution: **"playbook"** gets used for *both* the rubric and runbook shapes — our labels disambiguate it.) Tells you where the content belongs.
- **Reference / knowledge skill (resource-centric)** — the honorable-mention fourth shape: a skill whose body is **facts to read** (a small glossary, schema, or fact list). The right cheap tool when there are only a handful of terms to know; once the knowledge outgrows what fits comfortably in-context, use **RAG** ([L21](../L21/objectives.md), full course) instead. *A.k.a.* cheat-sheet, knowledge pack, reference doc.
- **Operating skill** — a higher-level skill that drives a task and may invoke other skills. (Contrast with a lower-level *shared* skill it calls.)
- **Shared (lower-level) skill** — a small skill invoked by more than one operating skill; the write-once-reuse-many node.
- **Sequential handoff** — composition where skill A produces an artifact that skill B consumes (A → B); a pipeline.
- **Skill dependency graph** — nodes = skills, edges = "invokes / hands off to"; the map of a skill system. Pipelines are chains; shared skills are fan-in nodes. At runtime an edge is a **`load_skill` call** from one skill's body to another; the agent discovers skills via **`list_skills`**.
- **`list_skills` / `load_skill`** — the two LangChain tools on the L04/L11 agent that implement JIT loading: `list_skills` returns all skills' `{name, description}` (discovery); `load_skill(name)` reads one skill's full body into context (load). The only skill-related things always in context are these two tool schemas.
- **Seam / handoff contract** — the artifact-shaped agreement between two composed skills: what the upstream must produce for the downstream to work.
- **Composition anti-pattern** — over-chaining, a shared "skill" that's really a tool, and description collisions.

## Main points the lecture should land

- **"Skill" is not one shape — name the archetype first.** Script-centric, principle-centric, procedure-centric. The archetype tells you *where the content goes* (a script, a rule list, an ordered procedure). Open here — it reframes L22's single skill as one point in a small space.
- **The archetype dictates the authoring.** Determinism → push it into a *script* and keep the markdown thin. Standards/taste → a *rule list* the agent judges against. A multi-step task → an *imperative runbook*. Writing prose where a script belongs (or a linear procedure where a rubric belongs) is the archetype-level version of L22's "wrong container" mistake.
- **Skills compose two ways, and both make a graph.** Sequential handoff (A → B, a pipeline) and a shared lower-level skill (A → C ← B, a dependency). The repo you're sitting in *is* a small skill system — `author-lesson-roadmap` hands off to `generate-materials-from-roadmap`. Composition is real, not hypothetical.
- **The seam is the design.** A handoff is only worth it if the boundary between the two skills is a real, stable *contract* (a well-defined artifact) — that's what makes each half separately loadable, testable, and re-runnable. A handoff added for tidiness, with a fuzzy seam, is just over-chaining.
- **JIT loading now happens across a graph, through two tools.** The agent calls `list_skills` to discover (surfacing all `{name, description}` for that turn) and `load_skill` to pull one body; the only always-on skill cost is the *two tool schemas*, regardless of how many skills exist or the graph's shape. Bodies load only along the path taken; a shared skill loads per-use, not per-caller. Depth costs loads and turns — wide-and-shallow is cheap and fails locally; deep chains concentrate risk.
- **Composition can make things worse — know the three ways.** Over-chaining (all overhead, no work), a shared "skill" that's really a tool (it should be *called*, not *read*), and description collisions (the model can't pick). A skill *system* is a win only when it's cheaper and clearer than the monolith it replaced.
- **This is the rehearsal for multi-agent.** Sequential handoff → a pipeline of subagents; a shared sub-skill → a shared worker; the dependency graph → the supervisor/worker call graph. (Full course: [L24](../L24/objectives.md). Mini cut: L23 lands the "compose capabilities into a system" idea; `MINI_WRAPUP.md` does the closing recap.)

## Common student confusions to watch for

- *"The archetypes are strict categories."* No — they're a spotting heuristic for the *primary* shape. Real skills borrow across archetypes (a runbook that calls a script). Classify to guide authoring, not to file paperwork.
- *"Composing skills means one giant skill that does everything."* The opposite — composition means *splitting* into skills with clean seams so each loads and runs independently. A mega-skill is what composition avoids.
- *"More handoffs = better decomposition."* Only if each seam is a real contract. Otherwise it's **over-chaining**: more load-overhead and more places to fail, for no reuse or testability gain.
- *"A shared skill is always the DRY win."* Only if it's genuinely *procedural know-how*. If every caller invokes it the same deterministic way with structured I/O, it's a **tool** wearing a skill's clothes — L08/L22 says call it, don't read it.
- *"If two skills sometimes both seem right, the model will just pick."* Not reliably — **description collisions** degrade selection across the whole set. Distinct, mutually-discriminating descriptions are a system-level design task, not a per-skill afterthought.
- *"A dependency graph means the skills run automatically like a program."* No — edges are *"may invoke,"* and each hop is still a model-driven JIT selection that can miss. The graph is a map of possible loads, not a guaranteed control flow.
- *"Deep chains are fine because each skill is small."* Depth costs a load (and often a turn) per hop and concentrates failure — a miss anywhere breaks the pipeline. Small skills don't make a deep chain cheap or robust.
