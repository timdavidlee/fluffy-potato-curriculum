## Where this lesson sits

By L12, students can *read* what their agent did: they instrument the hand-rolled L10 loop (`agent_loop.run(...)` → `RunResult(final_text, iterations, termination)`) to emit a **structured trace**, read that trace to reconstruct a run, locate a failure from the trace alone, and compare two traces of the same task to separate a real change from run-to-run noise. Crucially, L12's *tooled* half already put those traces into the cohort's **self-hosted Langfuse** instance — students have seen their runs in a real platform's UI. L12 framed itself as "produce the record"; L13 is **"judge the record."**

L13 is the turn from *observation* to *judgment*. Tracing tells you *what happened* on one run; evaluation tells you *whether it was good*, and — crucially — whether it is *still* good after you change something. The headline move is to stop asking "did this one run look right?" (the L12 eyeball diff) and start asking "do my agent's runs pass a fixed set of cases I defined in advance?" An **eval set** is that fixed set of cases plus a way to score them. It turns the ad-hoc failure-spotting of L12 into a repeatable practice.

This is also the lesson the PRD singles out as a *practice to carry forward*: "establish the practice that every later lesson will carry forward." After L13, every agent students build — the LangGraph shallow agent in L11, and beyond — should come with at least a minimal eval set. L13's job is to make that habit cheap enough to keep.

**How L13 teaches it — concept first, then the platform (same shape as L12).** Students first hand-build the *concept* of an eval — a case, a scorer, and a one-case run in ~15 lines of plain Python — so the mechanism is never magic. Then, immediately, they do the real work in **Langfuse**, the same self-hosted platform they already trace to: the eval set becomes a Langfuse **Dataset**, the runner becomes a Langfuse **Experiment**, the scorer becomes a Langfuse **score** (code-based *and* a managed LLM-as-judge), and the pass-rate / A-B / regression views are the Langfuse **run-comparison UI**. This mirrors L12 exactly: hand-build a trace, *then* meet Langfuse. The difference from L12 is that evaluation is where a platform earns its keep fastest — datasets, repeated experiment runs, and a managed LLM-as-judge are tedious to hand-roll and trivial in Langfuse — so L13 spends most of its minutes in the tool, not the hand-rolled sketch. **Decided:** L13 is *Langfuse-forward*: a thin hand-rolled concept beat establishes the vocabulary, then the lesson lives in Langfuse Datasets/Experiments/Scores. Large-scale LLM-judge rigor, RAG, and multi-agent eval are still deferred to L25.

> **Why the platform belongs here, not only in L25.** Every eval feature L13 needs — Datasets, Experiments, code-based scores, and a **managed LLM-as-judge** — is in the free, MIT-licensed, self-hosted Langfuse build the cohort already runs; none of it is paywalled. Teaching evaluation *through* the tool students will actually use (rather than a throwaway Python loop) is the realistic path, and it costs nothing extra to stand up because the instance is already live from L12. L25 then scales the *discipline* (LLM-judge done rigorously, retrieval metrics, multi-agent), it doesn't re-introduce the platform.

## Prerequisites

Students arriving at L13 should already be able to:

- Run the hand-rolled model→tool→model loop from L10 and read its `RunResult` (final text, iteration count, termination cause).
- Instrument that loop to emit a **structured trace** — an ordered list of typed events (model calls, tool calls with arguments, tool results with `is_error`, token usage, latency, termination) — and read it (L12, objectives 1 & 3).
- **Send a run to the cohort's self-hosted Langfuse instance and find it in the UI** (L12's tooled half). L13 builds directly on this: the same instance now stores datasets, experiments, and scores.
- Locate a failure from a trace alone and classify it by signature: tool error, wrong tool arguments, runaway loop, premature termination (L12, objective 2). These failure classes are exactly the targets L13's eval cases attack.
- Compare two traces of the same task and recognize that the loop is **non-deterministic**, so a single differing run proves nothing (L12, objective 4). L13 builds the disciplined version of this comparison.
- Reason about token usage and cost from L01 (cost estimation) and the per-call token/latency fields a trace records — L13's "eval cost" objective leans on this directly.

If a student cannot yet produce a trace *and see it in Langfuse*, redirect to the L12 labs first — L13 evaluates *traced runs* and does its scoring in the same Langfuse instance, so the L12 tooled step is a hard prerequisite here, not just the hand-rolled trace.

## Learning objectives

By the end of L13, a student should be able to:

1. **Build a minimal eval set for a tool-calling / hand-rolled-loop agent, and understand the three parts before reaching for the platform.** Concretely:
   - Define the three parts of an **eval set** and reuse the terms verbatim through the lesson: a **case** (an input the agent runs on plus what "good" means for it), a **scorer / check** (the function that turns one run into a pass/fail or a number), and the **runner** (the harness that runs every case and reports a summary).
   - Hand-build the concept once: write one `EvalCase`, one scorer, and a ~15-line one-case run in plain Python against the L10/L12 loop, so the mechanism is concrete and nothing about the platform is magic. This is the "concept" beat — kept deliberately tiny.
   - Then **map each part onto Langfuse** and reuse it for real: a case becomes a **Dataset item** (`input` + `expected_output`), the runner becomes an **Experiment** run over the dataset, and a scorer becomes a **score** attached to each run. Land that Langfuse's Dataset/Experiment/Score model *is* the case/runner/scorer vocabulary, just hosted and durable.
   - Distinguish **what** you can check and pick deliberately: the **final answer** (did `final_text` contain the right city?), the **trajectory** (did the trace show the expected tool sequence?), and **operational** signals (did it terminate `natural` rather than `max_steps`? under a token budget?). Land that an eval set can score the *answer*, the *path*, or both — and that for agents the path often matters as much as the answer. Each of these is expressible as a Langfuse score.

2. **Design eval cases that target failure modes already seen in traces from L12, stored as a Langfuse dataset.** Concretely:
   - Take the failure signatures students found in L12 traces — wrong tool arguments, a runaway loop ending in `max_steps`, an unrecovered tool error, premature `natural` termination on an incomplete answer — and turn each into a **regression case**: a Dataset item that reproduces (or used to reproduce) the failure, plus a code-based **score** that *fails* when the bug is present and *passes* when it's fixed.
   - Explain *why this ordering matters*: good eval cases come from observed failures, not from imagination. The trace is the source of truth for "what actually goes wrong"; L12 produced those traces (and put them in Langfuse) precisely so L13 has real failure modes to target. State the loop plainly: **trace a failure → add a dataset item that catches it → keep the case forever.**
   - Write at least one **trajectory-based** score (reads the run's trace, e.g. "did not call the same tool with the same args twice") and at least one **outcome-based** score (reads `final_text`), so students feel the difference between checking the path and checking the answer — both as Langfuse scores on the same experiment.
   - Recognize the limits of brittle checks: an exact-string match on `final_text` is fragile against harmless rewording; choose the *loosest check that still catches the bug* (substring, normalized number, "contains the right tool call"), and recognize when a more semantic check (an LLM-as-judge) is needed — which in Langfuse is a **managed evaluator you can turn on**, not something you hand-build (objective 4).

3. **Run an experiment across two models and compare pass rates in Langfuse to flag regressions.** Concretely:
   - Promote L12's eyeball trace-diff into a *repeatable* comparison: run the dataset as one Langfuse **Experiment** on version A (e.g. Sonnet 4.6) and another on version B (e.g. Haiku 4.5, or Sonnet after a prompt edit), and use Langfuse's **run-comparison view** to see which cases changed pass→fail (a **regression**) or fail→pass (a **fix**).
   - Confront **non-determinism head-on**: because the loop's path varies run-to-run, a single A-vs-B difference can be noise. Introduce the cheapest mitigation — run each case **more than once** and look at how often it passes (a pass *rate*, not a single pass/fail) — and read that rate as an aggregated score in Langfuse. Explain why a flaky case needs either a looser check, more samples, or a redesign.
   - Articulate the practice the PRD asks L13 to establish: an eval set is a **ratchet**. Once a case passes, a later change that breaks it is a regression you catch *before* shipping — and Langfuse keeps every past experiment run, so the ratchet is durable across days and people, not a table that scrolls off a terminal. Every subsequent lesson's agent should carry its dataset forward.

4. **Reason about eval cost and the scorer cost/judgment spectrum — ending on Langfuse's managed LLM-as-judge.** Concretely:
   - Account for the real cost of an eval run: each case is at least one full `agent_loop.run(...)`, which is *multiple* model calls (one per loop iteration) plus tool calls; running N cases × K samples multiplies that. Read the **per-call token counts straight off the Langfuse trace** for one experiment run (they are already there from L12) and multiply through — grounded in real data, not hand-waving — then cross-check against a back-of-envelope formula (`cases × samples × avg model-calls-per-run × per-call cost`).
   - Reason about the **sample-size trade-off**: more samples per case give a more trustworthy pass *rate* but cost linearly more. Pick a sample count deliberately for the lesson's toy set and name how you'd choose it for a real one.
   - Walk the scorer **cost/judgment spectrum** — exact assertion → fuzzy/automated check → **LLM-as-judge** → human review — ordered by cost and by how much judgment each exercises. Land it concretely by turning on **Langfuse's managed LLM-as-judge evaluator** on a quality no cheap check captures (e.g. "did the answer acknowledge the failure gracefully?"), and watch it score runs automatically. Flag its own costs out loud: it is a model call (tokens) and can be wrong, biased, or gameable.
   - Place **human review** at the expensive end (Langfuse's annotation/manual-scoring UI is where it happens) and decide, for a given quality, *which* scorer is worth its cost — a design choice, not a default.

5. **Establish evaluation-as-a-practice — a durable Langfuse dataset every later lesson carries forward.** Concretely:
   - State the discipline as a rule students will reuse: *when you build or change an agent, you add to or run your eval dataset.* Frame the L13 dataset (in Langfuse) plus the thin `common/evals.py` types as the seed that L11's LangGraph agent (and every agent after) plugs into.
   - Recognize what L13 is *not*: it is a *first pass* — a small dataset scored by a handful of checks — not a production eval system. L25 (Evaluation revisited) scales the same discipline to multi-step graphs, retrieval quality (precision@k / recall@k), LLM-as-judge done rigorously, and multi-agent systems, all on the same platform. Knowing where the first pass stops is part of the objective. **Decision:** the thin eval *types* — `EvalCase`, the `Scorer` protocol, `EvalResult`, plus a small **Langfuse bridge** (upload cases as a dataset, emit scorer results as Langfuse scores) — live in `common/evals.py` (alongside L12's `common/tracing.py` and the shared `common/tools.py`), so L11 and later lessons import them rather than redefine them.

## What an eval set is (vocabulary the lecture must establish)

Define these explicitly and reuse them verbatim through the labs and into L11/L25. Each maps onto a Langfuse concept — teach both names so students recognize the platform:

- **Eval set** → Langfuse **Dataset**. A fixed collection of cases plus the machinery to run and score them. The unit of "is my agent good, and is it still good?"
- **Case** → Langfuse **Dataset item**. One input the agent runs on, paired with a definition of "good" (an expected answer, an expected trajectory, an operational bound, or a mix). Stored as `input` + `expected_output`.
- **Scorer / check** → Langfuse **score** (code-based) or **evaluator** (managed LLM-as-judge). The function that turns one run (its `RunResult` + trace) into a verdict: pass/fail, a number, or a label. Ranges from an exact assertion to an LLM-as-judge to a human.
- **Runner / harness** → Langfuse **Experiment** (a run of the dataset). Executes every case (optionally K times), applies the scorers, and stores a summary you can re-open.
- **Regression** — a case that used to pass and now fails after a change. The thing an eval set exists to catch; visible as pass→fail in Langfuse's run comparison.
- **Pass rate** — for a non-deterministic agent, the fraction of K samples a case passes; replaces a single pass/fail once runs vary. Read as an aggregated score in Langfuse.
- **Outcome vs. trajectory scoring** — checking the *final answer* vs. checking the *path through the trace*. Agents often need both; both are just scores on the same experiment.

### Decided eval schema (thin types + Langfuse bridge)

**Decision:** L13 keeps a *tiny* set of hand-rolled types so students see the mechanism and so L11 can import a stable vocabulary — but the runner and the store are **Langfuse**, not a hand-rolled loop. The type names are a deliberately *approximate* match to how real eval platforms model evaluation, and line up with both **Langfuse** (dataset item with `input`/`expected_output` + score) and **LangSmith** (Example / Evaluator / `evaluate()`), so the names transfer to whichever tool students meet. Authored in `common/evals.py` (shared `common/` layer), consuming the `RunResult.trace` from `common/tracing.py`.

- **`EvalCase`** (Langfuse *Dataset item*; LangSmith *Example*): `id`, `inputs: dict` (the task/prompt), `reference_outputs: dict` (the expected answer and/or expected tool trajectory, e.g. `{"answer": "...", "expected_tools": ["calculator", "lookup"]}`). A helper uploads a list of these as a Langfuse dataset.
- **`Scorer`** (Langfuse *score function*; LangSmith *Evaluator*): a callable `(*, run: RunResult, example: EvalCase) -> EvalResult`. It reads `run.final_text` for **outcome** scoring or `run.trace` for **trajectory** scoring. A helper emits its result as a Langfuse score on the run.
- **`EvalResult`** (Langfuse *score*; LangSmith *EvaluationResult*): `key: str` (metric name, e.g. `"answer_correct"`, `"no_runaway"`), `score: float | bool`, `comment: str | None`.
- **The runner is Langfuse.** Instead of a hand-rolled `evaluate(cases, scorers, *, samples=K)` as the deliverable, students launch a Langfuse **Experiment** over the dataset (K samples per item) and read per-case scores and pass rates in the UI. A ~15-line `evaluate()` sketch is shown once, in the concept beat, to demystify what the platform does under the hood — then set aside.
- The **managed LLM-as-judge** is a Langfuse evaluator turned on in the UI/SDK, not a hand-written judge prompt in the harness. This is the single biggest reason L13 is Langfuse-forward: a rigorous-enough judge is one toggle, not a lesson's worth of prompt engineering.

## Main points the lecture should land

- **Evaluation is the judgment built on L12's observation.** Tracing told you *what happened*; evaluation tells you *whether it was good and whether it still is*. They are a pair — L12 produces the record (and puts it in Langfuse), L13 judges it (in the same Langfuse). Say it first, reusing the L12 framing students just saw.
- **You understand the primitive, then you use the platform.** A case, a scorer, and a run are ~15 lines of Python — see them once so nothing is magic. Then do the real work in Langfuse, because hand-rolling datasets, repeated experiments, and an LLM-judge is exactly the tedium a platform removes. (Same "concept first, then tooled" move as L12's trace → Langfuse.)
- **Good eval cases come from real failures, not imagination.** The traces from L12 are the source of cases. Trace a failure, add a dataset item that catches it, keep the case forever. An eval set grown from observed failures stays relevant; one invented up-front tends to test the wrong things.
- **An eval set is a ratchet against regressions.** Its real payoff is not the first green run — it's catching the day a prompt tweak or model swap silently breaks something that used to work. Langfuse keeps every experiment run, so the ratchet survives across days and teammates.
- **Non-determinism means you measure rates, not verdicts.** The agent loop varies run-to-run (L12's variance budget). A single pass or fail can be luck. The cheapest honest answer is a pass *rate* over a few samples — and a flaky case is itself a finding, not just noise to ignore.
- **Score the path, not only the answer.** For agents, *how* it got there matters: the right answer reached by a runaway, over-budget trajectory is still a problem. Trajectory scores (read the trace) catch what an answer-only check misses. This is why L12 came first.
- **Every scorer trades cost for judgment.** Exact assertions are cheap and dumb; humans are expensive and wise; the LLM-as-judge sits in between — and in Langfuse it's a toggle, not a build. Choosing a scorer is choosing a point on that curve for a given quality. There is no free, perfect, automatic check.
- **This is a first pass, on purpose.** A small dataset and a handful of scores in a platform you already run — not a production eval system. It establishes the *habit* and the *tool*; L25 scales the *rigor*. Naming that boundary keeps the lesson honest and the scope small.

## Common student confusions to watch for

- *"Evaluation is just testing."* Related, but unit tests assert deterministic outputs; agent eval scores a *non-deterministic* system, often on a *rate* and often on the *trajectory*, sometimes with a fuzzy or human judge. A `==` assertion is one scorer on the cheap end, not the whole picture.
- *"Langfuse is doing something magic."* It isn't — the concept beat showed the ~15-line version. A Dataset is a list of cases; an Experiment is a for-loop that runs and scores them; a score is a function's return value stored on a run. The platform makes it durable and comparable, not different.
- *"One run passed, so the case passes."* Not for a non-deterministic agent — one green run can be luck. Sample a few times and read the pass rate before believing it.
- *"Exact-match the final answer."* Usually too brittle — harmless rewording fails the check and trains you to ignore reds. Use the loosest check that still catches the bug; escalate to Langfuse's managed LLM-judge only when a cheap check genuinely can't express the quality.
- *"Checking the final answer is enough."* Often not — a correct answer reached via a broken, expensive, or runaway path is still a failure you want to catch. Score the trajectory when the path matters.
- *"More eval cases / more samples is always better."* Every case and every sample costs real model calls. A focused set targeting observed failures beats a huge set of invented ones, and sample count is a deliberate cost/confidence trade, not "max it out."
- *"The LLM-as-judge is a free way to check anything."* It's a scorer with its own cost and its own errors — it can be wrong, biased, or gameable, even as a one-toggle Langfuse evaluator. L13 turns one on to make the spectrum concrete; L25 unpacks what it can and can't reliably score.

## Bridge to L11 (and forward to L25)

In the mini cut, the next taught lesson is **L04 (Explicit graphs & workflows / DAGs)**, then **L11 (Shallow agents in LangGraph)** — and L11 is where the eval handoff lands. The handoff is the *practice* plus a *durable artifact*: when students rebuild the agent as a LangGraph graph in L11, they bring the **same Langfuse dataset** with them and run it as an experiment against the new implementation. (L04's deterministic workflow is itself trivially evaluable — a natural, cheap place to reinforce the habit even earlier.) That is the cleanest possible demonstration of L13's ratchet — *same dataset, different implementation, did anything regress?* — read directly in Langfuse's run comparison, and it makes "evaluate every agent you build" concrete the very next lesson. **Decided:** L11's lab reuses the L13 Langfuse dataset and the `common/evals.py` types/bridge against the LangGraph agent — the strongest reinforcement of the "carry it forward" objective.

Looking further ahead, **L25 (Evaluation revisited)** — outside the mini cut — extends this first pass to systems L13 deliberately doesn't touch: per-node vs. end-to-end metrics for multi-step graphs, retrieval quality (precision@k / recall@k) for RAG, LLM-as-judge done rigorously (rubrics, bias checks, judge-of-judges), and multi-agent evaluation — all on the same Langfuse the students have used since L12. L13 should explicitly signpost L25 as "where this scales," so students read L13's smallness as a starting point, not a ceiling.
