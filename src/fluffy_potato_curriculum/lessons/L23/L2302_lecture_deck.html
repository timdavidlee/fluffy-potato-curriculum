Slide 1.2

The organizing idea: center of gravity

  • Every skill bundles some mix of three things: instructions, scripts, and resources (facts/data). The archetype is just which one is the center of gravity.
  • Instructions split further: ordered steps (do this, then this) vs. judgment criteria (hold the work against these standards).

Slide 1.3

Why classify first

  • The archetype tells you where the real content belongs — and therefore how to author the skill well.
  • Mis-classifying produces a bad skill: prose describing a deterministic step, a linear procedure where a rule list belongs, retrieval stood up for six facts.
coral = the bad skill mis-filing produces; cyan = the correct classification. You classify not to file paperwork — the label picks your pen.

The three core archetypes

L23 · ARCHETYPES
Slide 2.1

Script-centric — the API / integration recipe

  • Center of gravity: a bundled script. The markdown is a thin wrapper; the real work runs.
  • The body says when to run it, the exact invocation, and how to read its output — nothing more.
  • Good when the operation is deterministic and better executed than reasoned through.

Slide 2.2

Principle-centric — the review / rubric

  • Center of gravity: a rule list. Principles or a checklist the agent applies as judgment — no single deterministic answer, no script that could replace it.
  • Written as criteria to hold work against, not a linear procedure.
  • Good when the value is consistent application of taste or standards.
the rule list — center of gravity
prefer plain functions
one behavior per test
annotate every public def
comment the why, not the what
the work — a diff under review

Slide 2.3

Procedure-centric — the operating-model runbook

  • Center of gravity: an ordered procedure. The skill drives a multi-step task end to end — numbered steps, decision points, a clear done-condition.
  • Written for an agent to follow, with deterministic sub-steps delegated to scripts.
  • Good when the value is doing the steps in the right order every time.

Slide 3.1

Reference / knowledge — resource-centric

  • Center of gravity: the resource. The body is facts to read — a small glossary, a schema, a handful of brand or product facts — not steps, criteria, or a script.
  • The cheap right answer when there are only a handful of terms to know: the facts just sit in context when the skill loads.
  • Taught as an aside, not one of the three you’ll author today — but a real, common shape.

Slide 3.2

When it outgrows a skill: RAG

  • Once the knowledge outgrows what fits comfortably in context, a reference skill is the wrong tool — that’s retrieval-augmented generation (RAG)’s job (full-course L21, not part of the mini cut).
  • Rule of thumb: a dozen definitions → a resource-centric skill; a document corpus → RAG.

Slide 4.1

Same format, different body shape

  • Every archetype uses the same SKILL.md format from L22 — frontmatter name + description, then a markdown body. What changes is the shape of the body.
SKILL.md — script-centric
name: get_order
description: when the user asks about an order
thin prose — when to run · exact invocation · how to read output
▸ get_order_api.py
SKILL.md — principle-centric
name: review_style
description: when reviewing a diff for style
prefer plain functions
one behavior per test
comment the why
SKILL.md — procedure-centric
name: author_roadmap
description: when drafting a lesson roadmap
1. read the docs in order
2. draft both artifacts
◇ cross-check ok? if not → 2
done when: report filed
identical ink-faint frontmatter, three cyan body shapes — the bodies miniaturize 2.1 / 2.2 / 2.3. No coral: all three are correct authoring.

Slide 4.2

The description is still the trigger — now across a set

  • Every archetype still needs a discriminating description (L22’s load-bearing craft point) — the sentence the model matches on to decide whether to read the skill at all.
  • L23 adds a constraint: descriptions must stay mutually distinct across the whole set of skills, or the model can’t reliably pick.
the set’s registry — each description says what it’s for, and implicitly what it’s not

The wrong-shape mistake

Slide 5.2

A spotting heuristic, not rigid boxes

  • Real skills borrow across archetypes: a runbook that calls a script for one step; a review skill that ends by running a formatter; a runbook that opens with a short reference glossary.
  • Classify the primary shape so the authoring choices follow — then borrow as needed.
