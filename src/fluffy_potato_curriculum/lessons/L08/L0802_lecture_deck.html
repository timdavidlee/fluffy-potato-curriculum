From making it work to making it good

  • L07 settled the mechanics — how a tool wires to a model call, the round-trip on the wire, the protocol. By the end of L07 a tool works.
  • L08 is a different thing: the protocol is fixed — the open questions are what to expose, how to name and shape it, and how to fail gracefully.
  • The model has no IDE, no autocomplete, no co-worker to ask — its whole understanding is the name, the description, the schema, and what comes back.

Why the LLM-consumer framing changes everything

  • A human client reads docs, tries a REPL, asks a co-worker. The model reads the tool spec once, at inference time, and decides on the spot whether and how to call it.
  • So every design choice optimizes for the model choosing and using the tool correctly on first read — not for the author's ergonomics.

Slide 2.1

Adding a tool is an architectural decision

  • The set of tools the model has is the set of actions it can take in the world. A tool is a decision about the agent's affordances, not a convenience.
  • And it has a cost shadow (L01): every tool eats system-prompt tokens, adds a per-turn decision, and adds another wrong-tool failure mode.

More tools ≠ more capable agent.

~4k tok10

the per-tool system-prompt token tax — a 5-tool agent the model can navigate beats a 20-tool agent it gets lost in.

Four signals a tool is warranted

SignalThe tell
Data the model can't have memorizedcurrent time, user-specific state, recent events
Precise computation the model is bad atlarge arithmetic, exact date math, hashing
A side effect outside the conversationsending email, writing to a DB, calling an external API
Verification against ground truthdatabase lookup, fact retrieval
  • The subtle one is computation the model is bad at: it will attempt it, confidently, and be wrong. The tool fills a gap the designer knows about, not the model.

L08 · TOOL DESIGN
Slide 2.3

Two signals no tool is needed

  • The model already knows it, reliably. "What is the capital of France?" — a tool here is pure overhead, and one more wrong-tool option to pick by mistake.
  • The round-trip costs more than the marginal accuracy. A tiny accuracy gain at the price of a full extra call + latency — skip it.
  • The discipline: run the task model-alone first, see where it actually fails, and add a tool only where the failure is real. Demo L0803 does this across three tasks; lab L0804 has you classify tasks and defend each call.

Slide 3.1

The name is read in isolation

  • The model sees the tool name without surrounding context — it must be descriptive, action-oriented, and unambiguous on its own.
  • Anti-patterns: cryptic abbreviations, overlapping names in one registry, and — worst — names that hide a side effect. A name that lies about what the tool does is a tool that will be misused.

Slide 3.2

The description is the most important field

  • Names are constrained by convention, schemas by types — the description is freeform prose: the only place to teach the model when (and when not) to use the tool, and its only training signal at inference time for selection.
  • Treat it like the system prompt of a tiny sub-policy — a weak one is the most common cause of an unused or misused tool. Demo L0805: same name, same schema, three descriptions → three behaviors.

L08 · TOOL DESIGN
Slide 3.4

Write for the model, not the human reader

  • The audience is the model's selection step at inference time — not the human reading the source. Code comments are for humans; tool descriptions are for the model.
  • Common confusion: "the description is for whoever reads the code." No — it's the prompt the model reads to decide. Write it short, concrete, examples included.
  • Overstating is as harmful as understating: promise billing + history when the tool returns only a name, and the model calls it expecting rich data and is confused by the thin response. Match the description to the implementation exactly.

The pieces of a good schema

LeverWhat it buys

Slide 4.3

Schemas validate shape, not truth

  • A tight schema catches malformed arguments (duration_minutes: 500 when the max is 240). It does not catch false ones — a well-formed but fabricated priya@example.com passes email-format validation even if no such user exists.
  • Exactly why errors (section 6) are the second line of defense: the schema rejects bad shape; the tool's runtime errors reject bad truth. Demo L0807 shows the model guessing a well-formed-but-fake value that only fails at lookup.

Slide 5.1

Distinguish three outcomes cleanly

  • A successful result, a successful call with no data (a lookup with no match — not an error), and an error — each must be tellable apart on sight.
the three-outcome fan — {"found": false} is neutral on purpose: not an error. 5.2 and 6.2 re-show this fan.

L08 · TOOL DESIGN
Slide 5.2

Why "no data" must not look like an error

  • If "no user found" returns the same shape as "lookup crashed", the model can't tell whether to retry (it might be transient) or to report (it's final). Conflating them produces undefined behavior.
  • A clean {"found": false} is a final, unrecoverable signal — the model reports back or asks for a better key, and does not retry. Demo L0809 shows this path explicitly.

Errors are part of the interface
An error is a prompt for the model's next turn — write it for the model, not the debugger.
24 / 37
L08 · TOOL DESIGN
Slide 6.1

An error is new context, not an exception

  • When a tool returns an error, the model's next turn is conditioned on that error text — exactly like any other context.
  • A well-shaped error teaches the model to fix its call; a bare exception teaches it nothing, so it guesses — often wrongly. A stack trace dumped into the tool result tells the model nothing actionable.

Errors should be informative for the model, not just for a human debugger.

Idempotency and side effects
The model retries on its own — a side effect it can't see is a side effect it can't avoid.
28 / 37
L08 · TOOL DESIGN
Slide 7.1

The model will retry on its own

  • When a result looks ambiguous, the model retries — regardless of whether retry is safe. You cannot rely on the model to know a call is dangerous to repeat.
  • So idempotency is a system concern, not just a tool-author concern. Even a perfectly described tool can be called twice by a confused model.

Slide 9.1

The design survives the packaging change

  • L09 introduces MCP (Model Context Protocol) — portable packaging that exposes the same tool to many clients without rewiring. It standardizes only the transport and discovery layer, not the design pressures.
  • Naming, descriptions, schemas, error shapes, idempotency apply unchanged — a poorly designed tool via MCP is still a poorly designed tool, just available to more clients. L09 leans on the L08 vocabulary throughout.
