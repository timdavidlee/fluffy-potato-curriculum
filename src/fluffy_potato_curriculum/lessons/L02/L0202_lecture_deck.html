Roles: who said what
The conversation is a list your code owns — and where you put content changes how the model weights it.
06 / 37
L02 · PROMPTING
Slide 2.1

A prompt is a list, and you own it

  • “The conversation” is a Python list of {role, content} messages your code assembles and re-sends every call — no hidden server-side memory (the L01 callback).
  • Three roles: system (out-of-band steering, always-true context), user (inputs from the caller), assistant (prior replies — or planted in-context examples, section 4).
messages = [ system user assistant user assistant … ]
The caller builds and re-sends this whole list every turn.

Slide 2.2

system = always-true, user = per-call

  • The rule to carry: durable identity-and-policy content goes in system; per-call task content goes in user.
  • Why it matters twice: reuse — a lean system message fronts many requests unchanged; and cost — system is re-sent every turn, so bloat is paid for forever (what prompt caching, L19, exists to fight).
system — always true
“You are a triage assistant. Answer in two short paragraphs…”
user — this call only
“My head has hurt for three days.”

Slide 2.3

The two mis-attribution footguns

  • Instructions buried in user get re-sent every turn — more tokens, and the policy now competes with the actual request for attention.
  • Per-call data crammed into system poisons reuse: the system message stops being always-true, so the next request inherits stale data.

Slide 2.4

The system message is weighted, not enforced

  • The model treats system as context with a privileged label — it tends to follow it, but it's not invulnerable to user-message instructions.
  • Anything near a security or policy guarantee belongs outside the model, in your code. The system prompt is not a sandbox.
  • Drop the misconception now: strongly weighted ≠ enforced.
Privileged label ≠ enforced.

Slide 2.5

Multi-turn, and the first-call vs. Nth-call axis

  • A multi-turn conversation just appends alternating user/assistant and re-sends the whole list — every turn re-bills the history (the L01 staircase).
  • First-call vs. Nth-call: a prompt that behaves on call 1 can degrade by call 5 as the model accumulates its own flawed prior turns.
  • Decide: continue when history is load-bearing; start fresh to dodge cost + drift.

Structured output: a negotiation
You ask for a shape; the model agrees to the contract but does not enforce it. So: ask, parse defensively, fail loudly.
12 / 37
L02 · PROMPTING
Slide 3.1

Why ask for structure at all

  • Structured output makes the reply programmatic, not conversational — a dict your code indexes, not a paragraph you read.
  • It's the precondition for almost everything later: evals (L13), tools (L07), pipelines all need machine-readable output.
  • Shapes you ask for: a JSON object, a fixed field list, or XML-ish tags like <answer>…</answer>.

Slide 3.3

The model agrees to the contract; it does not enforce it

  • Even told “JSON only,” the model can hand you any of these.
  • An enum is a contract, not a constraint — listing allowed values asks; it doesn't stop invention.
  • The mental model: ask for structure, parse defensively, fail loudly.

Slide 3.4

The defensive parser

  • Three steps: json.loads the whole reply; on failure regex the first {…} block and retry; on failure again raise loudly with the raw response.
  • Fail loudly, never silently. A silent {} fallback hides a bug that compounds downstream in agent loops (L10) and traces (L12).
  • Validate the contents: required keys present? enum in the allowed set? Decide null-tolerance on purpose.

Slide 3.5

Temperature, verbosity, and cost

  • Use low temperature (≈0) for extraction and classification — the single most-likely, repeatable answer, not creative variety.
  • Ask for a tight schema, no prose — this typically reduces output tokens. Structured output is often a cost win too (output is the pricey direction).
  • Keep request and reply shapes consistent — fewer degrees of freedom means fewer parse failures.

Few-shot examples: a precision tool
Showing, not teaching — example pairs that condition this one call's behavior, priced in tokens, where diversity beats volume.
19 / 37
L02 · PROMPTING
Slide 4.1

What few-shot is — and isn't

  • A few-shot prompt places a few input→output example pairs before the real input to show the desired output shape or behavior.
  • It's behavior conditioning, not teaching — capabilities don't change; you shift the in-context distribution. Frame it as showing, not teaching.
  • Drop the misconception: “few-shot teaches the model new things.” Nothing about the model is updated.

Slide 4.2

Two placements, one trade-off

alternating fake turns
user(ex1_in) assistant(ex1_out) user(ex2_in) assistant(ex2_out) user(real_in)
single user block
user("Examples:
ex1_in → ex1_out
ex2_in → ex2_out

Now classify: real_in")
  • Left: plant each example as a user then assistant message — the surprise is that assistant is the right home for fake “model said this” content.
  • Right: all examples as a labeled text block in one user message, then the real input.
  • Placement matters less than content — pick what your parser or cache prefers.

Slide 4.3

Diversity beats volume

4× the same easy case
P0-bug P0-bug P0-bug P0-bug
→ overfits: everything → P0
4 diverse examples
P0 P1 P2 wont-fix
→ generalizes across the set
  • A few contrasting examples outperform many similar ones — four labels beat four of the same label.
  • The failure to name: “all examples look like the easy case.” A uniform set makes the model overfit a surface pattern.
  • Distinguish few-shot for format (output shape) from for behavior (judgment) — both work, they fail differently.

Slide 4.4

Few-shot is priced on every call

  • Each example costs input tokens on every single call (the L01 callback) — few-shot is a real cost-vs-quality dial, not a free win.
  • It's also editable: add an example that resembles each failing input. That's its power and its trap — the list grows and per-call cost grows.

The task needs reasoning steps, not pattern-matchingchain-of-thought — L06

Slide 5.2

Extraction — pull fields out of text

  • Extraction is structured output pointed at “pull these fields.” Name the fields, ask for one JSON object, parse defensively.
  • Single-schema: a fixed set of keys every time. Mixed-schema: a list of heterogeneous items — validate each against the shape it claims.
  • The failure: the model invents a field or drops one from the text — the required-keys check catches it.

Slide 5.3

Classification — sort into a fixed label set

  • Structured output constrained to a label set — the place few-shot most earns its keep, because a team's label wording is often idiosyncratic.
  • Three flavours: flat (one of N), hierarchical (category then subcategory), multi-label (a JSON array).
  • An enum is a contract — validate label in ALLOWED, and for a taxonomy that the subcategory belongs to the category.

Slide 5.4

Ranking / recommendation — order a candidate list

  • Give the candidates ids and ask for the ids in order — never ask it to rewrite the candidates. Referencing by id keeps output tiny and the contract checkable.
  • Validate: every candidate id appears exactly once — no drops, dups, or inventions. The classic silent failure is one item quietly vanishing.
  • Optionally ask for a one-line justification per rank — useful, but it costs tokens and isn't what you validate.

Slide 5.5

Constrained generation — produce exactly N, within a bound

  • The one shape where you may want a higher temperature — but the constraint (count, length cap, format) must be explicit in the prompt and re-checked in code.
  • Ask for a JSON array of length N — a shape you can len()-check beats counting bullets by eye.
  • A constraint you don't check is one the model is free to miss — and it will, some runs.

Slide 5.6

Summarization and the transformation family

  • Summarize, rewrite, normalize, translate are one family: text in, transformed text out, meaning preserved. The system message carries the durable policy (audience, length, register).
  • The contract is softer — validate guardrails, not equality: length in bound, no invented facts. Failures: hallucinated additions and length drift.
  • Low temperature still helps for faithful summaries; lift it only for deliberately creative rewrites.

Slide 5.7

Picking the shape, and chaining shapes

  • Name the shape first. “This is a classification problem” tells you to reach for a fixed label set, few-shot for odd labels, and a label in ALLOWED validator — before you write a word of prompt.
  • Temperature by shape: near-0 for extraction / classification / ranking; higher only where generation wants variety.

Slide 6.1

The three levers and the catalog, reconnected

  • Roles decide where content lives; structured output decides the answer's shape (parse defensively); few-shot nudges behavior at a per-call token price.
