An LLM is a next-word predictor
One loop — predict the next token, sample, append, repeat. Every capability later in the course sits on top of this loop, not a separate "reasoning engine."

L01 · LLM & TOKENS
Slide 1.1

The one loop

  • An LLM does exactly one thing: given the text so far it predicts the next token, a sampler picks one, that token is appended, and the loop repeats until a stop signal or max_tokens.
  • "Writing a paragraph" is not a special mode — it is this single loop run a few hundred times. Everything later is built on this loop.

Why the pieces are sub-word tokens
The unit the model predicts is a token — often a whole word, often a fragment. And underneath, a token is really an integer.

L01 · LLM & TOKENS
Slide 2.1

The next piece is a token, not a word

  • The unit the model predicts is a token — often a whole common word, often a fragment. running may be one token; a rare or invented word splits into several.
  • So "predict the next word" is really "predict the next token." Everything downstream — window, cost — is counted in tokens.
running
common word — 1 token
flibbertigibbeting
rare word — 5 tokens
A common word costs one token; a rare or invented word fragments into many pieces that compose to spell it.

← → navigate
L01 · LLM & TOKENS
Slide 2.2

Why sub-word — not words, not characters

  • Not whole words? An unbounded vocabulary — every name, typo, hashtag, and new word would need its own entry, and you'd still miss tomorrow's words.
  • Not single characters? Sequences get very long and each step carries little signal — slow and wasteful.
  • Sub-word is the compromise: a fixed vocabulary of common chunks that compose to spell anything. Most tokenizers use BPE, learned from data — so different vendors split the same string differently.

Slide 2.3

A token is really an integer

  • The model never sees characters. It sees integer IDs — e.g. running → [16, 5143]. The readable "pieces" are a decoding of those IDs for your benefit.
  • The takeaway: the tokenizer is a chosen encoding, not a natural fact about language — which is exactly why real strings fragment the way they do.
running → running → [16, 5143]
text → sub-word pieces → integer IDs. The readable pieces are for you; the model only ever sees the integers.

Slide 3.2

The rule of thumb (and why it's only that)

  • Rule of thumb for plain English: ≈ 4 characters per token, or ≈ ¾ of a word per token.
  • It was calibrated on ordinary prose, so it breaks on the shapes above — often by 3× — because those strings were rare in training and are spelled out in pieces.

by tokens (~9): {"user_id": 12345}

Slide 3.3

Why this is the foundation for everything after

Every cost number and every window number later is downstream of the token count. Get tokens wrong and every estimate is wrong by the same factor.

What this is — and is not

Temperature: the knob on the distribution
Temperature reshapes the distribution before the sampler picks — low sharpens, high flattens. It doesn't change what the model knows.

L01 · LLM & TOKENS
Slide 5.1

Temperature reshapes before sampling

Temperature reshapes the distribution before the sampler picks: low sharpens it (top token dominates); high flattens it (more candidates compete). It doesn't change what the model knows.
temp 0 — one tall bar
temp ≈ 0.7 — a few compete
temp ≈ 1.0+ — many similar
The same distribution, three temperatures — a knob on how aggressively the sampler commits to the top token.

L01 · LLM & TOKENS
Slide 5.2

Temp 0 is low variance, not zero

  • Predict the effect: temperature 0 → near-identical answers run to run; temperature 1 → varied, sometimes creative, sometimes off.
  • But even at temp 0, output can differ: floating-point non-determinism, server-side batching, and tie-breaking between equally-likely tokens all leak through.
  • Frame it as low variance from day one — or a future eval that flakes will look like a bug when it's expected.

Slide 6.2

Reusable preambles — the sting in the tail

  • Once front-loading helps, you reuse the same block of instruction / persona / reference across many calls. That block is a preamble — the informal precursor to L02's system message.
  • The sting: with no server-side memory, that preamble is re-sent on every single call — re-read, re-counted against the window, re-billed. The very thing that made the output good is now overhead you carry on every call.

You just built up overhead on purpose. Section 8 is the bill.

Budgeting I — the context window
One hard ceiling on tokens per call — input and output combined. Everything competes for that one budget.

L01 · LLM & TOKENS
Slide 7.1

One shared budget

  • The context window is a hard ceiling on how many tokens a single call can involve — input and output combined.
  • Everything competes for it: the preamble / system message, every prior turn, tool definitions (L07 on), the current input, and the model's own response.
  • Anchor number: Claude Sonnet 4.6 has a 200,000-token window. (A 1M long-context variant exists; we use 200k as the course default.)

The input/output asymmetry

The whole chain in one breath

  • Predict the next token → the pieces are sub-word, so real strings fragment → a bigger model and better front-loaded context sharpen the prediction → temperature reshapes it → you reuse a preamble → and with no memory, you re-send and re-pay it against both the window and the bill.

Everything you front-load to make the model predict better is overhead you re-send, re-count, and re-pay on every call — so every prompt is a budget decision in tokens, dollars, and window space at once.
