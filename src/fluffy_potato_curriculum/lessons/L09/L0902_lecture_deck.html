The problem MCP solves
One tool, N clients, N hand-maintained integrations — the duplication is all glue, and MCP standardizes exactly that glue.

Slide 1.1

N clients, N integrations

  • Your L08 book_meeting is a function plus a tool definition — but every client registers it its own way: the Anthropic SDK wants tools=[{...}], a framework wants a decorator, an IDE plugin wants config.
  • The pain, stated precisely: a tool designed for one client requires N separate integrations to be reused across N clients. Three clients × five tools = fifteen hand-maintained integrations.

Slide 1.2

The duplication is glue, not logic

  • Look closely at those N integrations: the tool logic is identical in all of them — the same book_meeting(attendee, start, duration_minutes), name, description, schema, error shape.
  • Only the wrapping — how the client wants to be told about the tool — changes from client to client.
  • Back-reference L08: the design surface (name / description / schema / error) is exactly the part that should stay constant. MCP freezes a standard envelope around it.

If the wrapping is the only thing that varies, standardize the wrapping — and the duplication disappears.

Slide 1.5

What MCP does not change

  • MCP changes the packaging and the transport. It does not change the L08 design pressures: a bad name is still bad, a loose schema is still loose, a bare-stack-trace error is still useless — now over a wire instead of in-process.
  • Does MCP make design easier or harder? Neither. The L08 design work is identical; MCP just decides where the bytes go afterward.

The most repeated point of the lesson: MCP doesn't replace tool design — it's the L08 design, unchanged, now reachable over a wire.

Slide 2.3

The published tool list is an external API

  • Inline: the agent author owns the tool list. MCP: the server author owns it — the published list lands, at connect time, as runtime system prompt for every agent that connects. Give it the L08 care you'd give any tool description.
  • L08 again: more tools ≠ more capable agent. A server exposing 20 tools floods every client's prompt, dilutes attention, and raises the wrong-tool failure rate.

Slide 4.1

A server is a small program, not a microservice

  • A minimal MCP server is a script: define the tool function (your L08 design, unchanged), register it with a server object, pick a transport, run. Tens of lines, not a deployment.
  • Let go of another confusion: “an MCP server is a microservice.” It can be deployed like one over HTTP/SSE — but it can equally be a 50-line Python script launched as a stdio child process. Start small.
  • The build walkthrough L0906 shows the full skeleton for book_meeting (reused from L08) — shown as code, not run here (no mcp package in the course env).

Slide 4.3

Error-shaping must survive the boundary

  • In-process, an uncaught exception is a stack trace your code can catch. Across the boundary, an uncaught exception can crash the server — taking the tool offline mid-conversation: a transport-level failure, not a result the model can read.
  • The discipline: catch inside the tool, return a structured result — the same L08 {error, field, message} shape. The model gets actionable context; the server stays up. Error design is now load-bearing for availability, not just helpfulness.

Slide 4.4

You own the published surface

  • The server author owns the published tool list (slide 2.3) — so building a server means you decide what every connecting model sees.
  • Expose the one tool the server is for; resist the urge to bundle “just in case” tools.
  • Back-reference L08: a focused tool set is navigable; tool soup is not. A server is just a new place the same curation discipline applies.

When is MCP worth it?

L09 · MCP
Slide 5.1

The cross-process boundary is a feature and a tax

  • The boundary is a feature: isolation. The tool process can crash without taking the agent down, run with different permissions, and be replaced or upgraded independently.
  • The boundary is a tax: every call crosses a wire, errors can happen on either side, and you have one more thing to deploy and keep healthy.

This whole section is one question: is the feature worth the tax for this tool?

Slide 5.4

The decision is a gradient, not a gate

  • Many tools start inline and graduate to MCP once a second consumer appears. The decision is not permanent — it hinges on the number of consumers more than any other factor.
  • MCP early because it sounds “professional”? Resist: inline is the right default until reuse appears. Complexity that buys nothing is just complexity.
  • Weigh near-future consumers — but don't pay the tax for hypothetical reuse. Break-even is “one” for a tool you know will be reused, “many” for a one-off. The L0907 lab encodes this ledger as a decision function — pure Python, no mcp package.
The break-even chart, third beat: a gradient, not a gate — coral stays reserved for the cost line.

Slide 5.5

A word on security and credentials

  • One genuine reason the boundary is valuable: the server process can hold credentials the client never sees. The tool runs with its own permissions.
  • So a compromised or buggy agent can't directly reach the tool's secrets.
  • Hold onto this as a real benefit — but the deep dive (auth flows, secret storage) comes later in the course. Here it's one item on the benefit side of the ledger, not a full topic yet.

Slide 6.3

Bridge to L10

  • L10 builds a hand-rolled agent loop — model→tool→model in plain Python, across turns until the model decides to stop. The loop needs some tools to call.
  • The choice you can now defend: an inline tool (L08) or an MCP-served tool (L09), per section 5's trade-offs. L10 will likely use inline — but the loop is indifferent to where the tool lives; that indifference validates MCP.

You can now design a tool, package it for portability, and decide which packaging fits — next you'll put it in a loop and let the model call it until the task is done.

The chips are interchangeable — the loop is indifferent. Dashed = lands next lesson; no failure anywhere in this picture.
