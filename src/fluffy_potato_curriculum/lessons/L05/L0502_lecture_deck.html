Fixed edge vs. conditional edge

  • In L04 you only used fixed edges: add_edge("parse", "draft") — taken every time, no exceptions.
  • Now you add the conditional edge: a routing function reads state at runtime and returns the name of the next node.
  • Wired with add_conditional_edges("classify", route, {...}).
One fixed edge is always taken; a conditional edge (dashed) lets a runtime routing function fork to one of several branches that reconverge at END.

Slide 2.2

Determinism: the workhorse proof

  • Re-run the same ticket through your router and confirm the same branch executes every time.
  • The branch wording may vary (the model is still non-deterministic inside a node); the path doesn't.

A conditional edge is “runtime-chosen,” but it's a deterministic function of state — not a coin flip. That's what separates a workflow's routing from an agent's.

Slide 2.3

The fallback / default branch

  • Your router needs a defined behavior for every possible label — not just the expected ones.
  • When a classification doesn't cleanly match a branch, route it to a default general fallback rather than letting the routing function raise.
  • Treat this as a first-class design requirement, not an afterthought — a habit worth having before messier real-world classifiers.

Slide 3.1

The purest contrast with an agent

  • A conditional edge can branch on a value the user supplied — a menu choice, a form field: if state["user_choice"] == "escalate": return "human_review".
  • No model is involved in the routing decision at all. The user picked the path; you wired the options.
  • Here the input arrives as part of the initial state, then the graph runs straight through.
  • The interactive version — pause mid-run to ask, resume on the answer — needs interrupt + a checkpointer, in L17.

Section 4
When branching is enough
Reach for a router when the set of paths is known in advance. Reach for an agent only when the steps aren't enumerable ahead of time.
12 / 15
L05 · ROUTING
Slide 4.1

Router, not an agent, when paths are known

  • A router/switch is right whenever the set of possible paths is known in advance and picking among them is a classification, lookup, or user choice — cheap, predictable, testable.
  • Reach for an agent only when the model must decide which and how many steps to take, and that set isn't enumerable ahead of time.
  • Watch the common failure mode: building an agent to do what a two-branch router would do just as well — at a fraction of the cost and unpredictability.
