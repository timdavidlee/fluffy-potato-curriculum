Slide 1.1

From text-out to acting

  • L01–L06 only ever asked the model for text — even L06's reasoning and self-critique were text it produced and your code read. Nothing it said ever ran.
  • L07 is the first lesson where the model can act: request that your application run a function and feed the result back. This is the foundation of every agent later.
  • But it is not magic and not a new model capability — it is a protocol the model was trained to participate in.

L07 · TOOL CALLING
Slide 2.1

A tool is a function plus a description

  • Start with a plain function that does side-effect-free work — here a calculator(expression: str) -> str that evaluates arithmetic deterministically.
  • model.bind_tools([calculator]) builds the tool definition — a name, a prose description, and a JSON-Schema of the args — from the name, docstring, and type hints. The typed function is the schema.
  • The model never sees your Python function. It sees only that inferred definition. That is the entire contract.

Slide 2.3

The definition is a contract about shape

  • A tool definition tells the model "tools of this shape exist." Mirroring L06's <thinking> tags, it adds zero capability and makes zero guarantees about behavior.
  • The definition does not force the model to call the tool — it may answer directly instead.
  • It does not validate the arguments at generation time — the model can pass nonsense.
  • It does not stop the model from inventing a tool name that doesn't exist.

The application validates; the model proposes. Return here any time you expect the schema to enforce something.

Slide 2.4

Deciding to call is a reasoning step

  • Handing the model a tool is an offer, never a command. Whether it reaches for the tool is a sampling decision conditioned on the prompt, the conversation, and the tool's description.
  • That decision is itself a reasoning step — exactly what L06 was about. A vague tool the model can't tell when to use is a tool it will skip. Same skill, aimed at a new choice.
  • A look ahead you won't need yet: "what makes a tool worth adding, well-named, well-described?" is L08's whole job. For now, just notice the description visibly moves behavior.

Slide 3.2

The id ties request to result

  • The tool call in message 2 carries a unique id. The ToolMessage in message 3 sets its tool_call_id to that same id — that's how your app says "this result answers that request."
  • With a single call the id feels redundant. It stops being redundant the moment more than one call is in flight (a forward-link to L10's agent loop). Build the habit now.

Slide 3.3

The result rides in its own ToolMessage

  • This surprises everyone: the tool result does not go in an AIMessage, and it is not the human speaking either. It rides in a dedicated ToolMessage — a distinct tool role, your code speaking on the tool's behalf.
  • So the history carries three message types — human, ai, and tool — not just a human↔assistant back-and-forth.

The role a message carries marks its protocol position, not who is a person.

Slide 3.4

The model is stateless across calls

  • To produce message 4, the model was sent messages 1–3 and the full tool-definition list, again. It did not "remember" the tool from message 2.
  • The most common student misconception: "I told the model about the tool last turn, so it remembers." It does not. The schema is in the prompt every single time — and because you kept the same bind_tools handle, it was re-attached automatically.
  • Statelessness is why the next section is about cost — re-sending everything on every call is not free.

Tools cost tokens twice over

  • Because the model is stateless, a tool is paid for in two places on every turn: the definition is re-sent in the prompt of every request, and the result lives in the history for every subsequent turn.
  • This is L01's per-token cost shadow, now attached to tools. Nothing here is hypothetical — the demos print the token counts.

Slide 5.2

The model can hallucinate a call

  • The schema did not stop any of this at generation time: wrong argument types, a missing required argument, an extra invented argument, even a tool name that doesn't exist.
  • This is the L07 analogue of L06's tag-violation moment: one hallucinated call teaches you more than ten clean runs, because it proves the contract is best-effort.
  • The remedy is not a better schema (that's an L08 conversation) — it is that the application validates; the model proposes. Validation is not optional.

L07 · TOOL CALLING
Slide 5.3

Tool calls are not deterministic

  • The same prompt and the same schema can produce a call on one run and a skip on the next, with slightly different arguments each time.
  • This is why you validate every call rather than trusting a clean dry-run. A round-trip that worked at your desk can fumble in class.

That variance is the point, not a bug.

L07 · TOOL CALLING
Slide 5.4

A brief word on forcing a call

  • bind_tools accepts a tool_choice argument (auto / any / none / a specific tool) that biases the decision toward or away from a tool call.
  • But even forcing a call does not guarantee well-formed arguments — it only guarantees that some tool call is attempted. Forcing changes whether the model calls, not whether it calls correctly.
  • Worth knowing it exists, but the deeper tool_choice design conversation belongs to L08 — you'll get there.

Common confusions to retire

L07 · TOOL CALLING
Slide 6.1

The misconceptions, named

ConfusionCorrection
"The model runs the tool."The application runs the tool; the model only emits a request shaped like a tool call.
"If I define a tool, the model will use it."Not necessarily. Tool selection is a sampling decision conditioned on the prompt and the tool's description.
"Tool calls are deterministic."They are not. Same prompt + schema can call, skip, or pass different arguments across runs.
"The result is part of the assistant message."It is not. It rides in its own ToolMessage — your app speaking for the tool.
"More tools = better agent."More tools = more schema in every prompt, more mis-picks, more edge cases. (→ L08)
"I can force the model to always call this tool."tool_choice biases the decision, but forced choice still doesn't guarantee well-formed arguments.

Slide 7.1

From "it works" to "is it good?"

  • By the end of L07 you can build a tool-using exchange that works — wire one tool, trace one round-trip, and validate a hallucinated call. That is the mechanics.
  • L08 ("Designing good tools") takes those mechanics and asks the design questions: should this be a tool, what to name it, what the schema should look like, how it should report failure.

By the end of L07 you can make a tool call work; by the end of L08 you can argue whether the tool should exist at all.
