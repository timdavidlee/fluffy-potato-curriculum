Slide 1.1

Reconciliation, stated plainly

  • You have one receipt and a small ledger of records the company already booked. The question the agent answers: which record is this receipt, and do the amounts agree?
  • That sounds like a lookup — and the matching part is. The hard part is that the receipt almost never arrives in the ledger's shape. When nothing lines up, the right answer is to say so, not guess.
Both ends are legitimate — “no confident match” is a correct answer here, not a failure.

Slide 2.1

Where does the tool go? Apply the L08 test

  • Ask the L08 question: could the model do this reliably in its head? For “normalize a hotel folio's raw text and match it to a ledger row,” the honest answer is no — and that “no” is what warrants the tool.
  • So the split is deliberate: the tool does the brittle normalize-and-match; the model decides when to match, total, or give up. Draw that line before you write a line of code.

Slide 2.2

More tools make it a real decision

  • Give the agent three tools: find_matching_record, the reused calculator (total the items, check the variance), and a pre-built check_expense_policy (an LLM read of the free-form policy that cites the rule). Now it must choose which to reach for, and when.
  • That choice is the point. A one-tool agent has no decision to observe; a multi-tool agent leaves a trajectory — and a trajectory is the only thing worth tracing and evaluating.
Three cyan tools, one junction — the branch the agent takes (match, total, check the policy) is exactly what the trace records.

Slide 2.3

A tool can itself be a model call

  • Usually a tool is pure mechanism: find_matching_record — the one tool you write — normalizes and looks up, same input, same row, no model inside it. The tidy side of the L08 line you drew a slide ago.
  • check_expense_policy is provided, and it's a different animal: company policy is free-form prose (expense_policy.md), not a cap table — so the tool's body does a single model.invoke that reads the policy and cites the rule. The tool is a model call — policy is guidance to interpret, not a row to match.

Slide 3.2

What a failure has to become: a kept case

  • Finding the failure isn't the finish line — catching it forever is. Turn the trace into one EvalCase plus a Scorer, and reason about what it must do: fail when the bug is present, pass once it's fixed.
  • Read the trajectory, not just the answer — “no repeated identical matcher call” and “terminated natural, not max_steps.” That's the L13 loop on your own agent: trace → write a case → keep it.

Slide 4.1

“It ran once” is not done

  • The agent is non-deterministic — one green run proves almost nothing. So the bar isn't “it produced an answer.” The habit the whole mini course built: trace before you guess, eval or it's vibes.
  • The one sentence to carry into the build: the slice is done when it's traced and has one case that would catch its regression.

One new tool; the rest is assembly
