Slide 1.1

An agent is a graph with a cycle

  • The graphs you built in L04/L05 flowed forward and stopped — a DAG, where no node ever ran twice and you owned every edge.
  • An agent takes those same primitives — typed state, nodes, a conditional edge — and adds exactly one new thing: a back-edge from the tool node to the model node.
  • That single edge hands the model the decision of what runs next.

The back-edge is the cycle, and the cycle is the agent.

Slide 1.3

The three rules, up front

  • Rule 1 — the message-history invariant. The add_messages reducer makes every node append; the prebuilt ToolNode answers each .tool_calls entry with a matching ToolMessage. (section 2)
  • Rule 2 — termination is a branch of route. Natural termination is route returning END; recursion_limit is the built-in cap for everything else. (section 3)
  • Rule 3 — tool failures are messages, not exceptions. ToolNode(handle_tool_errors=True) converts a raised exception into a ToolMessage(status="error"). (section 4)

Slide 2.2

The state: the add_messages reducer

  • The agent's state is a TypedDict with one field: messages: Annotated[list, add_messages].
  • Back in L04, returning a state key overwrote it. The add_messages reducer flips that: each node's returned messages get appended instead of substituted.
  • That's why the conversation grows every turn — the agent node adds one AIMessage, and the tools node adds one ToolMessage per call.
  • add_messages also pairs messages by id, so history stays well-formed as it accumulates — half of Rule 1, with no bookkeeping from you.

Slide 2.3

The agent node

  • The agent node is just a plain function: state → {"messages": [reply]}.
  • Its body is a single line you already know from L07: model.bind_tools(tools).invoke(state["messages"]).
  • It returns the model's one AIMessage (maybe carrying .tool_calls) wrapped in the messages key, and the reducer appends it.
  • The model is not aware of the graph at all — every visit to agent is one independent round-trip. The loop lives in the edges.

Slide 2.4

The tools node = prebuilt ToolNode

  • ToolNode(tools) reads the last message's .tool_calls, runs each requested tool, and appends one ToolMessage per call.
  • This is the other half of Rule 1: after an AIMessage with .tool_calls, every call must be answered before the next model call — the #1 bug in hand-rolled loops.
  • Not magic: it's the L07 dispatch step, prebuilt. The same ToolNode runs L09's MCP tools unchanged — only the tool objects differ.

Slide 2.5

The wiring: route + the back-edge

  • Two edges finish the graph. route is the L05 conditional edge; the back-edge is a plain add_edge.
  • route(state) returns "tools" when the last message has .tool_calls, and END otherwise. Wire with add_conditional_edges("agent", route, {"tools": "tools", END: END}), then add_edge("tools", "agent") — the back-edge.
  • You write route by hand here. In L11 you'll meet LangGraph's prebuilt tools_condition — the same function, packaged.
  • Watch the cycle turn with graph.stream(task, stream_mode="updates"): {"agent": …} → {"tools": …} → {"agent": …} …

Nothing stops the cycle unless route says stop, or a cap fires.

Slide 3.1

Termination is a design decision

  • The model has no idea a graph exists. It'll keep emitting tool calls for as long as it thinks more of them help — nothing stops the cycle unless route says stop or a cap fires.
  • A cyclic graph with no cap isn't “minimal” — it's a runaway waiting to happen. Even a toy agent gets a recursion_limit.
  • Four termination conditions are worth knowing. You'll rely on the first two and sketch the other two as custom routing.

Four termination conditions

ConditionFires whenIn L10
Natural terminationthe agent node returns an AIMessage with no .tool_calls, so route returns END. The only condition meaning “the model thinks it's done.”rely on route
Recursion limitthe graph took more steps than recursion_limit allows and raises GraphRecursionError.rely on the built-in
Token budgetcumulative input+output tokens (or cost) cross a threshold.sketch as custom routing
Loop detectionthe model calls the same tool with the same arguments repeatedly without progress.sketch as custom routing

Natural termination is the only one that means “the answer is ready.” Every other condition means “we stopped it.”

Slide 3.3

recursion_limit is a safety net, not a correctness tool

  • The cap bounds the damage when something goes wrong — a runaway model, a tool that always looks unfinished, a prompt that never converges. What it does not do is make answers correct.
  • Hitting the cap is always a signal worth investigating. Either the task genuinely needs a higher cap, or the agent is misbehaving — treat GraphRecursionError as an alert, not noise.
  • recursion_limit lives on invoke, not the graph shape. It counts super-steps (node visits) — one tool per turn burns roughly two steps per cycle.

Slide 3.4

What to surface on a non-natural stop

  • When the graph stops because a cap fired — not because the model finished — you decide what the caller sees.
  • (a) Let GraphRecursionError propagate — the loudest option, right when a cap-hit is a true error your caller must handle.
  • (b) Route to a “last chance to summarize” node — friendlier output, at the cost of one more call.
  • Which one's right comes down to who consumes the result — a batch job usually wants (a); a chat UI might want (b).

Slide 3.5

Sketching the other two caps as custom routing

  • Here's the payoff of building route by hand: extra termination conditions are just more branches of the conditional edge. No new mechanism — you extend route.
  • Token budget — carry a running token sum in state (from reply.usage_metadata); have route return END once it crosses a threshold.
  • Loop detection — carry a short history of (tool_name, json.dumps(args)) tuples; return END if the last k are identical. Needs arguments and progress, not just call counts.
  • In a graph, termination is just another routing decision — the L05 skill, now pointed at END.

Slide 4.1

Two layers of failure handling

  • L08 taught the tool author what to return when something goes wrong — error-as-data, e.g. {"error": "row not found"}.
  • L10 teaches the graph what to do when the tool can't even return — when it raised, or handed back the wrong shape.
  • Those are different layers, and the graph owns the second one.

Slide 4.3

handle_tool_errors=True — one argument

  • The whole exception-to-message behavior is a single constructor argument: ToolNode(tools, handle_tool_errors=True).
  • Compare to handle_tool_errors=False: the exception escapes the node and kills the whole invoke. One buggy tool crashes the agent.
  • Flipping that single argument is the beat to watch: one word, and it's crash vs. recover.
  • With it on, a buggy tool degrades into a message instead of crashing — the model can retry, reach for a different tool, or apologize. Its call, not the graph's.

Slide 4.4

Don't dump tracebacks at the model

  • Feed the model a short, descriptive error string — "connection reset by peer", not a 40-line Python traceback.
  • Three reasons: tracebacks are token-expensive, they're noise the model can't act on, and they leak stack details (file paths, internals) you'd rather not expose.
  • The good news — ToolNode's default error text is already short: a class name plus one line.

Slide 4.5

Retries are a decision, not a default

  • ToolNode does not auto-retry — deliberately. Surfacing the error and letting the model decide is the default; it often knows whether a retry could even help.
  • Not all failures are alike: a 404 row not found will never succeed on retry; a 503 service unavailable might.
  • Blind retries waste tokens and can mask bugs — and with an idempotency-violating tool (charges a card, sends an email), auto-retry is actively dangerous.
  • If you do want auto-retry, add it deliberately, with its own budget — never as a reflex.

Slide 5.1

The graph didn't add behavior — it drew control flow as data

  • Put the hand-wired graph next to a bare while loop doing the same model→tool→model round, and they're the same skeleton.
  • agent = “call the model,” tools = “run the tools,” the conditional back-edge = “got tool calls? go again : stop.”
  • So the graph form didn't change what happens — it drew the control flow as data (nodes and edges). That's exactly what makes it packageable (L11) and inspectable (L12).

Slide 5.5

The minimum-viable trace, and the bridge ahead

  • By now you can read the returned messages and narrate the run turn by turn — agent, tools, agent, tools, agent — and say why it stopped. That narration is a minimum-viable trace.
  • Bridge to L11 — create_agent builds this exact graph in one line: MessagesState is your add_messages state, its tool node is the same ToolNode, its routing is the prebuilt tools_condition you wrote as route.
  • Bridge to L12 — that same stream, routed to a structured tracer: each node becomes a span.

An agent is a cyclic graph around a stateless model — until route returns END or recursion_limit fires; and when a tool breaks, ToolNode turns the break into a message, not a crash.
